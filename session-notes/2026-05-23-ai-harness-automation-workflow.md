# AI Agent Harness Automation Workflow Session Notes

Date: 2026-05-23  
Purpose: Capture this session as reusable source material for building an agent configuration, Skill creation process, and automation workflow around AI coding agents.

## 1. Session Goal

This session started as planning for a 4-hour lecture about Claude/Codex configuration, Superpowers, Skills, and automation workflows. The direction evolved into a more general framework:

> Build a practical AI agent harness that combines prompt patterns, memory files, Skills, Subagents, Hooks, MCP/tool connections, evaluation, and recurring automation.

The resulting lecture materials are not only presentation content. They are also a blueprint for creating a reusable automation workflow for AI coding agents.

## 2. Core Model

The working model settled on this layered structure:

```text
Prompt Layer
  Persona, few-shot examples, output format, XML/delimiters

Memory / Context Layer
  CLAUDE.md, AGENTS.md, auto-memory, project memory, references

Skill Layer
  Reusable procedural playbooks, SKILL.md, references, scripts, assets

Agent Layer
  Subagents, reviewer agents, security agents, research agents, agent teams

Automation Layer
  Hooks, Stop hooks, PostToolUse hooks, SessionStart hooks, Permission hooks

Tool Layer
  MCP, GitHub, Browser, DB, Slack, Figma, filesystem tools

Evaluation Layer
  test, lint, typecheck, build, reviewer, LLM-as-judge, human review

Loop / Schedule Layer
  repeated checks, scheduled monitors, recurring documentation or PR review jobs
```

This is the conceptual basis for the future automation workflow.

## 3. Important Distinctions

### Prompt vs Harness

Prompt engineering is not separate from harness engineering. It is the innermost layer.

```text
One-off instruction -> Prompt
Repeated instruction -> CLAUDE.md / AGENTS.md
Repeated procedure -> Skill
Repeated role -> Subagent
Must-run behavior -> Hook
External capability -> MCP / Tool
Completion proof -> Evaluation
Repeated operation -> Loop / Schedule
```

### Memory vs Context Engineering

Memory and context engineering are related but not identical.

```text
Memory = what should be retained
Context engineering = what retained information should be loaded right now
```

Memory types discussed:

```text
Session context
  The current conversation and immediate working memory.

Project memory
  Claude Code: CLAUDE.md
  Codex: AGENTS.md

User/global memory
  User-level instructions or global memory files.

Auto-memory
  Files such as feedback_*, project_*, reference_* may appear in memory folders.
  These are likely generated by auto-memory systems, plugins, or custom agent workflows.

Skill memory
  Skill references, scripts, and assets loaded only when needed.

External memory
  Docs, qmd indexes, vector stores, RAG, databases, or knowledge graph outputs.
```

### CLAUDE.md vs AGENTS.md

For the lecture and workflow, treat these as parallel concepts:

```text
Claude Code -> CLAUDE.md
Codex       -> AGENTS.md
```

Both are explicit project memory files. They should contain stable project rules, test commands, coding conventions, and verification expectations.

### Auto-memory Files

Files like these are not the default `CLAUDE.md` structure:

```text
feedback_*
project_*
reference_*
```

They likely come from one of:

```text
Claude Code auto-memory
Plugin-specific memory systems
Custom memory workflows
Previous agent-generated memory organization
```

The automation workflow should treat these as categorized memory fragments, not as project governance files.

## 6. Automation Workflow To Build From This

The future workflow should help an agent or user create a project harness in this sequence:

```text
1. Intake
   Gather project type, language, framework, test commands, deployment context, risk level.

2. Prompt Layer Setup
   Define default response style, output formats, review persona, and few-shot examples.

3. Project Memory Setup
   Generate or update CLAUDE.md / AGENTS.md with stable project rules.

4. Context Strategy
   Decide what belongs in always-on memory, Skill references, external docs, qmd, or RAG.

5. Skill Creation
   Create one or more Skills for repeated procedures.
   Examples:
   - issue-triage
   - code-review
   - verify-change
   - document-update

6. Subagent Setup
   Create role-specific agents.
   Examples:
   - code-reviewer
   - security-reviewer
   - research-agent
   - test-reviewer

7. Hook Setup
   Start with safe echo hooks.
   Then add formatter/lint/test hooks.
   Add Stop Hook once the verification criteria are clear.

8. Tool / MCP Setup
   Connect only needed tools.
   Separate read-only and write-capable tools.
   Avoid excessive tool descriptions and context overhead.

9. Evaluation Setup
   Define completion evidence.
   Include:
   - changed files
   - commands run
   - outputs observed
   - residual risk
   - human review notes

10. Loop / Schedule Setup
   Add recurring automation only after the single-run workflow is reliable.
```

## 8. Candidate Skills To Generate

### issue-triage

Purpose: Analyze a bug report or feature request before implementation.

Recommended steps:

```text
1. Restate the request in one sentence.
2. Identify likely affected files.
3. Identify risks and edge cases.
4. Propose the smallest safe implementation plan.
5. List verification commands.
```

### verify-change

Purpose: Ensure changes are verified before completion.

Recommended steps:

```text
1. List changed files.
2. Identify relevant verification commands.
3. Run the narrowest useful checks.
4. Report command output summary.
5. State residual risk.
```

### code-review

Purpose: Review local changes for defects.

Recommended priority:

```text
1. Correctness bugs
2. Regressions
3. Missing tests
4. Risky assumptions
5. Maintainability issues
```

## 9. Candidate Subagents

### code-reviewer

Role: Find correctness issues, regressions, missing tests, and risky assumptions.

Expected output:

```text
Findings first.
Severity ordered.
File/line references where possible.
No praise before findings.
```

### security-reviewer

Role: Check authentication, authorization, secret handling, injection risks, and dangerous file/network operations.

### research-agent

Role: Collect current documentation or official references without polluting the main working context.

### test-reviewer

Role: Check whether test coverage matches changed behavior and edge cases.

## 10. Candidate Hooks

Start safe:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'File changed. Run targeted verification.'"
          }
        ]
      }
    ]
  }
}
```

Then project-specific:

```text
JavaScript/TypeScript: pnpm lint, pnpm test, pnpm typecheck
Python: ruff check ., pytest
Go: go test ./...
```

Advanced hooks to consider:

```text
SessionStart
  Load dynamic context.

PostToolUse
  Run formatter/linter/test after edits.

PermissionRequest
  Route risky permissions to human review.

Stop
  Prevent completion before verification.

PostCompact
  Re-inject compacted context reminders.
```

```text
5. Should Skills be generated as local project files or user-level reusable skills?
```


The generator should be conservative:

```text

Keep always-on memory short.
Put long examples in references.
```
