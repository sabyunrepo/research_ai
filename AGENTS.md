## Explanation And Practice Boundary

- 설명 슬라이드는 Act별 개념, 김아이 서사, 회사 업무 비유, 다음 실습으로 넘어가는 브릿지만 다룬다.
- 실습은 설명 슬라이드가 끝난 뒤 별도의 실습 슬라이드 또는 실습 UI에서 진행한다.
- Act 0은 실습이 없다. Act 1부터 실습 계약을 `practice-plan.md`와 `practice-spec.json`에서 별도로 관리한다.

## Slide Source Rules

- 슬라이드는 발표 진행의 1차 기준이다. 발표자가 별도 대본을 보지 않아도 화면 구조만 보고 말할 순서를 알 수 있어야 한다.
- 한 슬라이드에는 하나의 핵심 설명만 담는다. 10초짜리 전환 슬라이드라도 필요하면 분리한다.
- 슬라이드 수에는 사전 상한을 두지 않는다. 발표 흐름과 이해도를 위해 10초 또는 30초만 쓰는 짧은 메시지 슬라이드도 자유롭게 추가한다.
- 헤드라인은 `개요`, `배경`, `실습` 같은 제목이 아니라 그 장에서 말해야 하는 핵심 결론을 완결된 문장으로 쓴다.
- 각 슬라이드는 `헤드라인 메시지 -> 시각 앵커 -> 예시/근거 -> 다음 장 브릿지` 구조를 우선한다. 앵커 3개는 화면 밀도를 낮추기 위한 기본값일 뿐, 필요한 설명을 누락하라는 제한이 아니다.
- 한 메시지를 이해하는 데 4개 이상의 앵커가 필요하면 4개 이상을 사용할 수 있다. 다만 앵커 수가 많아 글자가 작아지거나 발표자가 한 장에서 두 가지 설명을 해야 하면, 슬라이드를 2장 이상으로 나눈다.
- 슬라이드 점검 시에는 `앵커가 3개를 넘었는가`보다 `필요한 앵커가 3개 제한 때문에 빠졌는가`, `한 장에 너무 많은 앵커가 있어 가독성이 떨어지는가`, `한 헤드라인 아래 서로 다른 메시지가 섞였는가`를 우선 확인한다.
- 어려운 용어는 먼저 회사 업무 비유로 설명하고, 실제 하네스 용어는 보조 라벨로 연결한다.
- 비유와 실제 용어를 단순히 `A = B`로 붙이지 않는다. 새 개념을 설명할 때는 `현실 비유 -> 작동 원리 -> 문제가 생기는 조건 -> 실제 하네스 용어 -> 실습에서 확인할 증거` 순서로 연결한다.
- 예: Act 1의 데스크 비유는 "데스크는 김아이가 일을 하기 위한 자료만 올려놓는 공간"으로 시작한다. 관련 없는 자료가 올라오면 의사결정이 혼란스러워지고 작업이 느려진다. AI에서 이 공간은 대화/세션을 시작하면서 쌓이는 내용이며, 이를 `Context`라고 부른다. 이 흐름을 거친 뒤에야 `Context Curation`을 "데스크에 올릴 자료를 고르는 일"로 연결한다.
- 다른 Act에도 같은 설명 구조를 적용한다. 예를 들어 매뉴얼 비유는 `반복 업무 절차 -> 절차가 없을 때의 문제 -> Skill`, 검문소 비유는 `완료 전 확인 -> 증거가 없을 때의 문제 -> Hook/Evaluation`으로 연결한다.
- 사용자가 특정 Act만 지시한 경우 그 Act만 확장한다. 다른 Act에는 원칙이나 TODO성 방향만 남기고, 실제 슬라이드/목업 확장은 요청받은 Act 범위 안에서만 진행한다.
- 슬라이드 화면에는 긴 대본을 넣지 않는다. 발표자 힌트, 전환 질문, 강조 문장은 source의 speaker note 영역에 둔다.

## Visual Asset Rules

- 이미지는 장식용으로 만들지 않는다. 모든 이미지는 강의 내용을 설명할 수 있는 teaching role을 가져야 한다.
- 이미지는 실제로 발표자가 가리키며 설명할 수 있는 장면, 비유, 비교, 업무 흐름, 검증 구조를 담아야 한다.
- 이미지 요구사항은 먼저 `asset-pack.json`에 `id`, `teachingRole`, `semanticRequirements`, 생성 프롬프트, 스타일 제약, 설명 앵커로 정의한다.
- 슬라이드는 이미지 경로를 직접 반복하지 않고 `visualAssetId`로 참조한다.
- 여러 이미지를 한 장의 시트로 생성해 나눠 쓸 경우, 원본 시트와 crop region을 기록하고 최종 참조는 실제 분할 파일을 기준으로 한다.

## Mockup Rules

- HTML 목업은 리뷰용이다. 하네스 산출물의 원천은 markdown/json 소스다.
- 목업은 슬라이드 흐름, 이미지 배치, 텍스트 크기, Act 간 중복 여부를 확인하기 위해 만든다.
- 목업에서 발견한 문제는 최종 HTML만 손보지 말고 content source, slide spec, asset pack 중 원인 레이어에 반영한다.
