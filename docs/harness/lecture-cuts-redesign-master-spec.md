# Lecture Cuts Redesign Master Spec

Generated: 2026-05-27
Last Updated: 2026-05-28

## Purpose

이 문서는 `lecture-cuts` 발표 프로젝트를 새 구성으로 다시 작성하기 위한 개선 적용 스펙이다.
기존 58장의 내용을 원재료로 사용하되, 새 발표는 `신입 AI 김아이` 서사와 반복 실습 루프 중심으로 재구성한다.

현재 문서의 확정 범위는 Act 1부터 Act 6까지다. Act 0은 도입부로 남겨 두고, 본문 실습 구조는 Act 1~6에서 닫는다.

## Global Redesign Constraints

- 대상은 개발자가 아니라 일반인 수강생이다. 전문 용어는 회사 업무 비유로 먼저 설명한 뒤 실제 용어로 연결한다.
- 전체 강의 시간은 실습을 포함해 4시간이다. 발표, 실습, 공유, 쉬는 시간을 모두 이 범위 안에서 설계한다.
- 실습은 설명을 쉬게 만드는 장치가 아니라 학습의 중심이다. 4시간 중 최소 90분 이상은 수강생이 직접 입력, 선택, 실행, 재시도하는 시간으로 배정한다.
- 이미지 자산은 슬라이드 장식이 아니라 발표자가 그림을 보며 설명할 수 있는 teaching artifact여야 한다. 각 자산은 어떤 말을 유도하는지 `teachingRole`과 `explanationAnchors`로 기록한다.
- 섹션별 이미지 자산은 `asset-pack.json`에 먼저 정의한다. 여러 이미지를 한 번에 생성한 시트가 있으면 원본 시트와 crop region 정보를 남기고, 슬라이드는 `visualAssetId`로 필요한 자산을 참조한다.
- 슬라이드 생성 지시는 XML 스타일 경계로 분리한다. `instruction`, `screenContent`, `speakerNavigation`, `assetRequirement`가 섞이면 안 된다.
- projector slide에는 `screenContent`와 발표 내비게이션에 필요한 짧은 cue만 보인다. 원문, 생성 지시, 이미지 프롬프트, 긴 speaker note는 presenter review와 spec에만 남긴다.
- Act 7은 만들지 않는다. 최종 적용 실습을 마지막에 몰아넣지 않고, 각 Act 안에서 설명과 실습을 함께 닫는다.
- 각 Act 실습은 게임처럼 보이는 활동이 아니라 실제 AI 작업 흐름의 축소판이어야 한다.
- 각 Act 실습은 단발 제출이 아니라 반복 개선 루프여야 한다.
- 수강생은 최소 2회 이상 다시 시도할 수 있어야 한다.
- 실습은 `입력 -> 산출물 생성 -> 검증 -> 피드백 -> 재시도 -> 비교 -> 베스트 저장` 흐름을 따른다.
- 슬라이드는 `docs/harness/lecture-cuts-slide-structure-v2.md`의 헤드라인, 3앵커, 브릿지 원칙을 따른다.

## Audience And Duration

이 강의는 AI 도구를 업무에 쓰고 싶지만, 개발 배경이나 에이전트 시스템 지식이 없는 일반인을 기준으로 설계한다.

수강생이 이미 알고 있다고 가정하지 않는 용어:

- 세션
- 컨텍스트
- 프롬프트
- Skill
- Agent / Subagent
- MCP
- Hook
- Evaluation
- LLM judge
- Harness

각 용어는 다음 순서로 소개한다.

```text
회사 업무 비유 -> 실제 용어 -> 실습에서 보이는 화면 -> 왜 필요한지
```

4시간 강의의 기본 시간 배분은 다음 기준으로 잡는다.

```text
도입과 기대값 정렬: 10-15분
Act 1-3 입력면 실습: 90-110분
휴식: 10분
Act 4-6 하네스 장치 실습: 90-110분
전체 정리와 질의응답: 15-25분
```

시간 설계 원칙:

- 한 번에 15분 이상 설명만 이어가지 않는다.
- 각 Act는 짧은 설명 뒤 바로 눈에 보이는 실습으로 들어간다.
- 일반인 대상이므로 로컬 실행형 실습은 Act 5 하나로 제한하고, 나머지는 웹 실습 또는 체크리스트형 실습으로 둔다.
- 발표자가 모든 개념을 깊게 설명하려고 하지 않는다. 수강생이 직접 결과를 보고 이해하는 쪽에 시간을 쓴다.

## Lecture Scope

이 강의는 `프롬프트 잘 쓰는 법`만 다루지 않는다. 최종 범위는 AI에게 일을 안정적으로 맡기기 위한 하네스 엔지니어링 전체 구조다.

다루는 주제:

- 정보 선별: AI에게 일을 맡기기 전에 무엇을 알려줘야 하는지 판단한다.
- Prompt: 이번 한 번의 업무 지시를 목표, 조건, 완료 기준으로 구성한다.
- Context / Session: 지금 작업에 필요한 자료와 오래된 가정을 구분한다.
- Memory / CLAUDE.md: 매번 반복되는 회사 규칙을 항상 유지되는 규칙판으로 분리한다.
- Skill / Superpowers: 반복 업무 절차를 재사용 가능한 업무 매뉴얼로 만든다.
- Agent / Subagent: 구현, 리뷰, 조사, 보안 같은 판단 공간을 역할별로 나눈다.
- Tool / MCP / Permissions: 역할별로 필요한 외부 도구와 권한을 좁혀 연결한다.
- Hook / Evaluation / LLM Judge: 완료를 말이 아니라 검증 증거와 점수로 판단한다.
- Harness Engineering: 위 요소를 하나의 반복 가능한 작업 시스템으로 연결한다.

다루지 않는 것:

- 특정 벤더 기능 사용법만 나열하는 도구 소개 강의
- 모든 프롬프트 패턴을 외우는 프롬프트 모음 강의
- Claude Code 설정 파일 전체를 세부 구현까지 설명하는 레퍼런스 강의
- MCP 서버 구현 튜토리얼
- Hook, Evaluation, LLM judge의 전체 프로덕션 운영 설계

```text
역할과 도구를 나눠도, 완료를 말로만 믿으면 다시 흔들립니다. 마지막에는 검증 게이트가 필요합니다.
```

### Decision Rule

각 항목은 다음 형식으로 확정한다.

```text
1. 목적
2. 포함할 내용
3. 제외할 내용
4. 슬라이드/실습에서 보이는 형태
5. 완료 기준
```

항목 하나가 확정되기 전에는 다음 항목의 세부 구현으로 넘어가지 않는다.

## Act 0: 오늘의 약속

### Status

Act 0은 확정된 도입부 구성이다. 4시간 일반인 대상 강의의 기대값을 정렬하고 Act 1 실습으로 진입한다.

### Objective

Act 0의 목적은 일반인 청중에게 `오늘 4시간 강의가 단순한 프롬프트 작성법이 아니라, AI에게 일을 안정적으로 맡기는 시스템을 직접 만들어 보는 시간`이라는 기대값을 먼저 맞추는 것이다.

Act 0은 도구명과 세부 구현을 설명하지 않는다. 대신 김아이라는 신입 AI 비유, 오늘의 학습 목표, 실습 방식, 최종적으로 얻어갈 산출물을 짧게 제시한다.

### Core Message

```text
오늘의 목표는 AI를 더 똑똑하게 만드는 것이 아니라, AI가 흔들려도 다시 올바른 일로 돌아오게 만드는 구조를 배우는 것입니다.
```

### Teaching Promise

Act 0에서 청중에게 약속하는 결과는 다음 세 가지다.

```text
1. AI에게 일을 맡길 때 무엇을 알려줘야 하는지 구분한다.
2. 프롬프트, 컨텍스트, Skill, Agent, Hook이 각각 어디에 쓰이는지 이해한다.
3. 직접 실습하면서 점수, 결과물, 검증 로그로 내 지시가 왜 잘됐는지 또는 왜 흔들렸는지 확인한다.
```

### Opening Story

김아이는 새로 입사한 AI 신입사원이다. 김아이는 빠르고 똑똑하지만 회사 상황, 팀 규칙, 완료 기준을 저절로 알지는 못한다.

상사가 이렇게 말하면 김아이는 추측으로 빈칸을 채운다.

```text
이거 알아서 잘 만들어줘.
```

이번 강의는 김아이에게 일을 잘 시키는 개인기만 배우는 시간이 아니다. 김아이가 계속 좋은 결과를 내도록 `업무 지시`, `작업대`, `업무 매뉴얼`, `팀 배치`, `자동 검문소`를 하나씩 만드는 시간이다.

### Slide Role In Act 0

Act 0은 3-4장 정도의 슬라이드로 구성한다.

권장 슬라이드 흐름:

1. `Hero Slide`: 오늘 우리는 김아이라는 AI 신입사원에게 일을 맡긴다.
2. `Promise Slide`: 오늘 얻어갈 것은 프롬프트 문장이 아니라 AI 업무 시스템이다.
3. `Map Slide`: 업무 지시, 작업대, 매뉴얼, 팀 배치, 검문소를 순서대로 만든다.
4. `Bridge Slide`: 첫 번째 실패는 대충 시킨 업무 지시에서 시작된다.

### Detailed Slide Composition

Act 0의 슬라이드는 발표자가 별도 스크립트를 보지 않고도 진행할 수 있어야 한다. 각 슬라이드는 `헤드라인 = 첫 문장`, `3개 앵커 = 말할 순서`, `시각 자료 = 기억 장치`, `하단 브릿지 = 다음 장 전환 멘트`로 구성한다.

#### Slide 0-1: 오늘 우리는 AI 신입사원 김아이에게 일을 맡깁니다

역할:

```text
강의 세계관 진입
```

화면에 들어갈 내용:

```text
헤드라인:
오늘 우리는 AI 신입사원 김아이에게 일을 맡깁니다.

3개 앵커:
1. 빠르다
2. 똑똑하다
3. 하지만 회사 맥락은 모른다

하단 브릿지:
문제는 능력이 아니라, 일을 맡기는 방식입니다.
```

발표자가 슬라이드만 보고 말할 흐름:

```text
김아이라는 AI 신입사원을 소개한다.
김아이는 빠르고 똑똑하다고 말한다.
하지만 회사 상황, 고객, 기준, 완료 조건은 저절로 알지 못한다고 짚는다.
오늘은 AI 성능이 아니라 일을 맡기는 방식을 다룬다고 연결한다.
```

시각 자료:

```text
김아이 캐릭터 또는 신입사원 책상 이미지
빈 노트, 노트북, 물음표 3개
업무 교육 느낌을 유지하고 지나치게 장난스럽게 만들지 않는다.
```

CSS/구성 지침:

```text
헤드라인은 44-52px 수준으로 크게 둔다.
3개 앵커는 아이콘 + 짧은 단어로 둔다.
하단 브릿지는 화면 아래 고정된 한 줄로 둔다.
텍스트는 강의장 뒤에서도 읽히도록 줄 수를 줄인다.
```

## Act 1: 김아이의 첫 실패

### Objective

Act 1의 목적은 수강생이 `AI가 이상하게 답했다`를 모델 탓으로만 보지 않고, `업무 지시와 작업 조건이 비어 있어서 AI가 빈칸을 추측했다`고 이해하게 만드는 것이다.

Act 1은 도구명, 하네스 전체 지도, CLAUDE.md, Skill, Hook 같은 용어를 앞세우지 않는다. 먼저 신입사원 김아이의 실패 장면을 보여 주고, 수강생이 직접 애매한 지시의 결과를 확인하게 한다.

### Core Message

```text
김아이는 똑똑하지만, 빈칸이 있으면 회사 맥락을 추측으로 채웁니다.
```

### Narrative Setup

김아이는 새로 입사한 똑똑한 AI 신입사원이다. 상사는 김아이에게 이렇게 말한다.

```text
반응형 카드 컴포넌트 하나 만들어줘.
```

문제는 이 지시 안에 다음 정보가 없다는 점이다.

- 누가 쓰는 카드인가
- 어느 화면 크기에서 봐야 하는가
- 어떤 콘텐츠가 들어가는가
- 어떤 디자인 제약이 있는가
- 완료 기준이 무엇인가
- 결과를 어떤 형식으로 보고해야 하는가

김아이는 멈춰서 질문하기보다 그럴듯한 기본값을 상상한다. 그래서 결과물은 있어 보이지만, 실제 요구와는 어긋난다.

### Slide Role In Act 1

Act 1의 슬라이드는 설명보다 장면을 먼저 보여 준다.

권장 슬라이드 흐름:

1. `Story Slide`: 김아이는 똑똑하지만 회사 맥락은 모른다.
2. `Contrast Slide`: 대충 시킨 지시는 그럴듯한 실패를 만든다.
3. `Practice Slide`: 목표를 보고 어떤 정보를 같이 줘야 하는지 고른다.
4. `System Slide`: 좋은 정보, 조건부 정보, 방해 정보를 점수로 본다.
5. `Bridge Slide`: 좋은 지시는 어떤 정보를 채워야 하는가.

### Practice Name

```text
실습 1: 김아이에게 무엇을 알려줘야 할까요?
```

### Practice Goal

수강생이 목표를 보고 김아이에게 어떤 정보를 함께 줘야 하는지 고른다. 이 실습은 프롬프트를 직접 쓰는 활동이 아니다. Act 2가 프롬프트 작성 실습이므로, Act 1은 `업무를 맡기기 전에 필요한 정보와 불필요한 정보를 구분하는 감각`을 만든다.

이 실습은 너무 쉬운 함정 선택지를 피한다. `알아서 잘 해줘`처럼 명백히 나쁜 선택지는 쓰지 않는다. 대신 실무에서 실제로 줄 법하지만 지금 목표에는 과하거나, 순서가 이르거나, 오히려 맥락을 흐리는 애매한 선택지를 섞는다.

### Learner Task

수강생은 3개의 업무 목표를 차례로 보고, 각 목표마다 김아이에게 함께 주입할 정보를 다중 선택한다.

각 문항의 선택지는 세 종류로 나뉜다.

```text
필수 정보: 없으면 김아이가 추측해야 하는 정보
조건부 도움 정보: 상황에 따라 도움이 되지만 지금은 과하거나 순서가 이른 정보
방해 정보: 그럴듯하지만 현재 목표를 흐리거나 오래된 가정을 섞는 정보
```

### Practice Questions

#### Question 1: 반응형 카드 컴포넌트

목표:

```text
김아이에게 강의 신청 페이지에 들어갈 반응형 카드 컴포넌트를 만들게 해야 합니다.
어떤 정보를 함께 줘야 할까요? 모두 고르세요.
```

선택지:

```text
[ ] 카드가 들어갈 페이지의 목적과 사용자가 눌러야 할 행동
[ ] 모바일 375px, 데스크톱 1024px 기준의 배치 조건
[ ] 카드에 들어갈 제목, 설명, 가격, CTA 문구
[ ] 기존 브랜드 색상과 버튼 스타일
[ ] 텍스트 overflow, hover/focus 상태, 반응형 확인 기준
[ ] 결과를 HTML/CSS 코드블록으로 나눠 달라는 출력 형식
[ ] 작년에 쓰던 랜딩 페이지의 전체 디자인 시스템
[ ] 경쟁사 카드 UI 스크린샷 10장
[ ] "프리미엄하고 세련된 느낌"이라는 톤 설명
[ ] 모든 브라우저와 모든 디바이스에서 완벽해야 한다는 요구
[ ] 나중에 붙일 결제 기능의 데이터 구조
[ ] 회사 소개 페이지의 전체 카피 문구
```

채점 의도:

- 필수 정보는 페이지 목적, 화면 기준, 실제 콘텐츠, 스타일 제약, 검증 기준, 출력 형식이다.
- 경쟁사 스크린샷과 톤 설명은 조건부 도움 정보다. 적게 쓰면 도움이 될 수 있지만, 요구사항보다 스타일 모방을 우선하게 만들 수 있다.
- 전체 디자인 시스템, 모든 디바이스 완벽 요구, 결제 기능 데이터 구조, 회사 전체 카피는 지금 목표에 비해 과하거나 범위 밖이다.

#### Question 2: 결제 버튼 오류 수정

목표:

```text
김아이에게 결제 페이지의 "신청하기" 버튼이 가끔 눌리지 않는 문제를 조사하고 수정하게 해야 합니다.
어떤 정보를 함께 줘야 할까요? 모두 고르세요.
```

선택지:

```text
[ ] 문제가 발생한 브라우저와 화면 크기
[ ] 버튼을 누르기 전 사용자가 거친 단계
[ ] 관련 파일 또는 컴포넌트 이름
[ ] 최근 수정된 결제 페이지 변경 내역
[ ] 재현 성공/실패 여부를 확인할 기준
[ ] 수정 후 실행해야 할 확인 절차
[ ] 결제 도메인 전체 ERD
[ ] 6개월 전 A/B 테스트에서 사용한 버튼 문구
[ ] 아직 확정되지 않은 다음 분기 결제 플로우 초안
[ ] 고객센터에 들어온 결제 관련 문의 전체 원문
[ ] "사용자가 불편하지 않게 자연스럽게 고쳐줘"라는 요구
[ ] 버튼 색상을 더 눈에 띄게 바꾸자는 의견
```

채점 의도:

- 필수 정보는 재현 조건, 사용자 단계, 관련 파일, 최근 변경, 확인 기준, 수정 후 검증 절차다.
- 고객센터 문의 전체 원문은 일부 힌트를 줄 수 있지만, 그대로 주면 노이즈가 커질 수 있다.
- 결제 도메인 전체 ERD와 다음 분기 플로우 초안은 지금 버그 수정 범위를 넓힌다.
- 버튼 색상 의견은 클릭 불가 버그의 원인 분석을 흐릴 수 있다.

#### Question 3: 발표자료 슬라이드 다듬기

목표:

```text
김아이에게 AI 하네스 강의의 첫 섹션 슬라이드를 더 이해하기 쉽게 다듬게 해야 합니다.
어떤 정보를 함께 줘야 할까요? 모두 고르세요.
```

선택지:

```text
[ ] 청중 수준과 강의 시간
[ ] 첫 섹션에서 수강생이 얻어야 할 학습 목표
[ ] 유지해야 할 핵심 비유와 용어
[ ] 바꿔도 되는 범위와 건드리면 안 되는 범위
[ ] 슬라이드 글자 크기와 발표 화면 제약
[ ] 완료 후 확인할 기준: 작은 글씨, 중복, 흐름 단절 여부
[ ] 전체 4시간 강의의 모든 발표 스크립트
[ ] 참고할 만한 다른 강사의 슬라이드 30장
[ ] 아직 검증되지 않은 새 실습 아이디어 전체
[ ] "일반인이 이해할 수 있게 쉽게"라는 단독 지시
[ ] 발표자가 개인적으로 좋아하는 디자인 무드보드
[ ] 이후 섹션에서 다룰 Hook과 Evaluation의 상세 구현 계획
```

채점 의도:

- 필수 정보는 청중 수준, 학습 목표, 유지할 비유/용어, 수정 범위, 화면 제약, 확인 기준이다.
- 다른 강사의 슬라이드와 디자인 무드보드는 참고가 될 수 있지만 과하면 현재 섹션의 메시지를 흐린다.
- 전체 발표 스크립트와 이후 섹션 상세 구현은 첫 섹션 수정에는 과하다.
- `쉽게`라는 단독 지시는 방향은 맞지만 검증 가능한 기준이 아니다.

### Practice Loop

Act 1 실습은 단발 제출로 끝나지 않는다.

필수 루프:

1. `Round 1`: 반응형 카드 컴포넌트 목표에서 필요한 정보를 고른다.
2. `Feedback 1`: 좋은 정보, 조건부 정보, 방해 정보를 색상과 점수로 확인한다.
3. `Round 2`: 결제 버튼 오류 수정 목표에서 필요한 정보를 고른다.
4. `Feedback 2`: 문제 유형이 바뀌면 필요한 정보도 달라진다는 점을 확인한다.
5. `Round 3`: 발표자료 슬라이드 개선 목표에서 필요한 정보를 고른다.
6. `Final Score`: 3문항 총점과 상사 타입 타이틀을 확인한다.
7. `Review`: 내가 자주 고른 과한 정보와 자주 놓친 필수 정보를 확인한다.

Act 1의 목적은 완벽한 프롬프트를 만드는 것이 아니다. 완벽한 프롬프트 작성은 Act 2에서 다룬다. Act 1은 목표를 보고 `김아이에게 무엇을 알려줘야 하고 무엇은 지금 주면 안 되는지`를 구분하는 데 집중한다.

### Result Screen

결과 화면은 다음 패널을 가진다.

```text
결과
- 3문항 총점
- 상사 타입 타이틀
- 문항별 점수

선택 분석

김아이가 추측하게 될 빈칸
```

### Scoring In Act 1

Act 1의 점수는 수강생을 평가하기 위한 것이 아니라, 정보 선별 습관을 보여 주는 진단 장치다.

각 선택지는 다음 가중치를 가진다.

```text
필수 정보: +10~+20
조건부 도움 정보: +2~+6
과한 정보: -3~-8
범위 밖 또는 오염 정보: -8~-15
```

3문항 합산 점수에 따라 상사 타입을 부여한다.

```text
90~100점: 김아이의 좋은 멘토
75~89점: 방향은 잡아주는 상사
55~74점: 빈칸을 남기는 상사
35~54점: 추측을 부르는 상사
0~34점: 최악의 상사
```

Act 1에서는 `총점`보다 `놓친 필수 정보`, `과하게 넣은 정보`, `김아이가 추측하게 될 빈칸`을 더 크게 보여 준다.

### Feedback Tone

피드백은 수강생을 비판하지 않는다. 김아이가 왜 그렇게 행동했는지 냉정하게 설명한다.

좋은 피드백:

```text
경쟁사 카드 UI 스크린샷은 참고가 될 수 있지만, 지금 단계에서는 요구사항보다 스타일 모방을 우선하게 만들 수 있습니다.
먼저 페이지 목적, 화면 기준, 완료 기준을 고정한 뒤 참고 예시를 1~2개만 주는 편이 안전합니다.
```

나쁜 피드백:

```text
틀렸습니다.
이 정보는 필요 없습니다.
```

### What The Instructor Says From The Slide

슬라이드에는 긴 대본을 넣지 않는다. 발표자는 다음 흐름만 보고 말한다.

```text
헤드라인:
김아이는 빈칸이 있으면 추측으로 일을 시작합니다.

3앵커:
1. 필요한 정보
2. 그럴듯하지만 과한 정보
3. 김아이가 추측할 빈칸

브릿지:
이제 이 정보를 실제 프롬프트 문장으로 바꿔보겠습니다.
```

### Acceptance Criteria

Act 1 설계는 다음 조건을 만족해야 한다.

- 3개의 다중선택 문항이 있다.
- 각 문항은 실무에서 실제로 줄 법한 애매한 선택지를 포함한다.
- 명백히 나쁜 함정 선택지에 의존하지 않는다.
- 수강생이 놓친 필수 정보와 과하게 넣은 정보를 확인할 수 있다.
- 3문항 총점과 상사 타입 타이틀이 표시된다.
- 결과가 Act 2의 프롬프트 작성 실습으로 자연스럽게 연결된다.
- Act 2의 `좋은 업무 지시` 필요성이 자연스럽게 열린다.

### Out Of Scope For Act 1

Act 1에서는 다음을 다루지 않는다.

- 완벽한 프롬프트 공식
- CLAUDE.md
- Skill
- Agent/Subagent
- Hook
- Evaluation 전체 구조
- LLM judge의 상세 rubric

이 항목들은 뒤 Act에서 각각 다룬다. Act 1은 오직 `빈칸이 있으면 AI가 추측한다`를 실제 결과로 확인하는 데 집중한다.

## Act 2: 좋은 업무 지시

### Objective

Act 2의 목적은 수강생이 좋은 프롬프트를 이론으로 외우는 것이 아니라, 직접 입력하고 점수를 올려가며 결과물이 바뀌는 것을 확인하게 만드는 것이다.

Act 1에서 수강생은 애매한 지시가 실패를 만든다는 것을 확인했다. Act 2에서는 같은 작업을 더 구체적으로 지시하면서, 프롬프트 품질이 실제 HTML/CSS 결과물 품질을 어떻게 바꾸는지 본다.

Act 2는 `프롬프트 잘 쓰는 팁`을 나열하는 구간이 아니다. Act 1에서 실패한 김아이에게 같은 일을 다시 제대로 맡기는 구간이다.

### Core Message

```text
좋은 프롬프트는 김아이에게 목표, 조건, 완료 기준을 함께 주는 업무 지시서입니다.
```

### Teaching Flow

Act 2의 강의 흐름은 다음 순서를 따른다.

1. `Act 1 결과 회수`: 김아이가 실패한 이유를 모델 성능이 아니라 빈칸 많은 업무 지시로 다시 짚는다.
2. `좋은 업무 지시의 6칸 소개`: 좋은 프롬프트는 긴 글이 아니라 필요한 칸이 채워진 지시서라고 설명한다.
3. `나쁜 지시와 좋은 지시 비교`: Act 1의 짧은 지시와 개선된 지시를 나란히 보여 준다.
4. `직접 입력 실습`: 수강생이 같은 일을 다시 지시한다.
5. `결과 생성`: 김아이가 실제 HTML/CSS 카드 컴포넌트를 만든다.
6. `점수와 검증 로그 확인`: 어떤 칸이 채워졌고 어떤 칸이 빠졌는지 확인한다.
7. `재시도`: 점수와 힌트를 보고 프롬프트를 고쳐 다시 실행한다.
8. `결과 비교`: Act 1 결과, Act 2 Attempt 1, Act 2 Attempt 2/3을 비교한다.
9. `Act 3 연결`: 매번 모든 맥락을 프롬프트에 다시 써야 하는지 질문하며 컨텍스트와 규칙판으로 넘어간다.

### Slide Role In Act 2

Act 2는 7-9장 정도의 슬라이드로 구성한다. 각 슬라이드는 `docs/harness/lecture-cuts-slide-structure-v2.md`의 슬라이드 타입을 따른다.

권장 슬라이드 흐름:

1. `Story/Bridge Slide`: 좋은 업무 지시는 결과를 바꾼다.
2. `Concept Slide`: 좋은 프롬프트는 긴 글이 아니라 빠진 칸이 없는 지시서다.
3. `Contrast Slide`: `반응형 카드 만들어줘`에는 중요한 빈칸이 빠져 있다.
4. `Concept Slide`: 목표, 대상, 조건, 기준을 채우면 김아이의 추측이 줄어든다.
5. `Contrast Slide`: 같은 요청도 지시서가 바뀌면 결과물이 달라진다.
6. `Practice Slide`: 김아이에게 반응형 카드 컴포넌트를 다시 맡긴다.
7. `System Slide`: 점수는 정답 판정이 아니라 빠진 칸을 보여 주는 진단 도구다.
8. `Practice/System Slide`: 점수를 보고 프롬프트를 고치면 결과물이 개선된다.
9. `Bridge Slide`: 매번 모든 맥락을 다시 쓰는 것은 비효율적이다.

### Required Prompt Ingredients

Act 2에서 소개하는 좋은 업무 지시의 6칸은 다음과 같다.

```text
1. 목표
2. 대상/사용 상황
3. 화면 크기 또는 반응형 조건
4. 제약 조건
5. 완료 기준
6. 보고/출력 형식
```

각 칸은 점수 항목이면서 동시에 발표자가 설명할 3앵커의 재료다. 한 슬라이드에 6칸을 모두 길게 설명하지 않는다. 먼저 전체 지도를 보여 주고, 실습 화면에서 빠진 칸을 하나씩 체감하게 한다.

### Practice Name

```text
실습 2: 김아이에게 제대로 업무 지시하기
```

### Practice Goal

수강생이 반응형 카드 컴포넌트 제작 프롬프트를 직접 작성하고, 점수와 결과물을 보면서 여러 번 개선한다.

Act 2는 전체 강의의 핵심 실습이다. 수강생은 여기서 프롬프트가 추상 문장이 아니라 산출물 품질을 바꾸는 업무 지시서라는 것을 확인해야 한다.

### Learner Task

기본 과제:

```text
김아이에게 강의 신청 페이지에 들어갈 반응형 카드 컴포넌트를 만들라고 지시해보세요.
```

수강생은 자유 입력폼에 프롬프트를 작성한다.

Act 2에서는 좋은 프롬프트의 정답을 먼저 주지 않는다. 수강생이 먼저 쓰고, 점수와 결과를 본 뒤 보완하게 한다.

### Before / After Example

Act 2의 비교 슬라이드와 실습 힌트는 다음 예시를 기준으로 한다.

Before:

```text
반응형 카드 만들어줘.
```

After:

```text
강의 신청 페이지에서 사용할 카드 컴포넌트를 HTML/CSS로 만들어줘.
375px에서는 세로로 쌓이고, 1024px 이상에서는 3열 그리드로 보여줘.
텍스트가 카드 밖으로 넘치지 않아야 하고, 버튼 hover/focus 상태를 포함해줘.
결과는 HTML과 CSS 코드블록으로 나눠서 보여줘.
```

After는 모범답안이 아니라 방향을 보여 주는 예시다. 실제 실습에서는 수강생이 자기 표현으로 다시 작성한다.

### Practice Loop

Act 2는 최소 3회 시도를 권장한다.

필수 루프:

1. `Attempt 1`: 수강생이 처음 작성한 프롬프트를 실행한다.
2. `Result`: 김아이가 만든 HTML/CSS 카드 컴포넌트를 확인한다.
3. `Score`: 프롬프트 점수와 빠진 항목을 확인한다.
4. `Feedback`: 개선 힌트를 확인한다.
5. `Attempt 2`: 빠진 항목을 보완해 다시 실행한다.
6. `Compare`: 이전 결과와 현재 결과를 비교한다.
7. `Attempt 3`: 완료 기준과 출력 형식을 보완해 최종 버전을 만든다.
8. `Save`: 가장 좋은 프롬프트와 결과물을 저장한다.

Act 2 화면에는 시도 번호, 이전 점수, 현재 점수, 점수 변화, 저장된 베스트 버전이 보여야 한다.

### Result Screen

결과 화면은 네 개 탭으로 구성한다.

```text
결과
- 현재 점수
- 김아이가 만든 카드 컴포넌트 미리보기
- HTML/CSS 결과 요약
- 이전 시도 대비 점수 변화

채점 기준
- 목표: 있음 / 없음 / 부분적
- 대상/사용 상황: 있음 / 없음 / 부분적
- 반응형 조건: 있음 / 없음 / 부분적
- 제약 조건: 있음 / 없음 / 부분적
- 완료 기준: 있음 / 없음 / 부분적
- 출력 형식: 있음 / 없음 / 부분적

개선 예시
- 빠진 항목을 보완한 예시 문장
- 전체 프롬프트 개선안

검증 로그
- prompt-goal-check: pass
- audience-context-check: partial
- responsive-condition-check: fail
- completion-criteria-check: missing
- output-format-check: pass
```

Act 2의 결과 화면은 Act 1보다 더 적극적으로 `점수 올리기`를 유도한다. Act 1이 실패 진단이라면 Act 2는 개선 루프다.

### Mechanical Scoring Logic

1차 채점은 기계적으로 한다.

점수 기준:

```text
목표 있음: 20점
대상/사용 상황 있음: 15점
화면 크기 또는 반응형 조건 있음: 15점
제약 조건 있음: 15점
완료 기준 있음: 20점
보고/출력 형식 있음: 15점
```

판정 예시:

```text
목표 있음
- "만들어줘", "구현해줘", "작성해줘"만 있으면 partial
- 구체적 산출물 이름이 있으면 pass
- 예: "강의 신청 페이지용 카드 컴포넌트"

대상/사용 상황
- "누가", "어디에서", "어떤 화면에서" 중 하나 이상 있으면 partial
- 사용 맥락이 명확하면 pass

반응형 조건
- mobile, desktop, 375px, 768px, 1024px, grid, stack 같은 조건이 있으면 pass

제약 조건
- 색상, 레이아웃, 접근성, 라이브러리 제한, HTML/CSS만 사용 같은 조건이 있으면 pass

완료 기준
- overflow 없음, hover/focus, 버튼 상태, 텍스트 줄바꿈, 375px 확인 등 검증 가능한 기준이 있으면 pass

출력 형식
- HTML/CSS, 코드블록, 설명 포함 여부, 보고 형식 요구가 있으면 pass
```

기계 채점은 완벽한 품질 평가가 아니라 빠진 칸을 찾는 진단 장치다.

### LLM Judge Role

LLM judge는 2차 평가만 맡는다. 총점을 단독으로 결정하지 않는다.

LLM judge 평가 항목:

```text
- 지시가 초보자도 이해할 만큼 구체적인가
- 요구사항끼리 충돌하지 않는가
- 결과물이 검증 가능한 기준을 포함하는가
- 불필요하게 장황하지 않은가
- 김아이가 질문 없이 작업을 시작할 수 있는가
```

LLM judge 결과는 다음처럼 표시한다.

```text
llm-judge: needs improvement
reason: 반응형 기준은 있지만 완료 기준이 검증 가능한 문장으로 충분히 분리되어 있지 않습니다.
```

### Feedback Tone

피드백은 `좋다/나쁘다`가 아니라 `다음 시도에서 무엇을 추가할지`로 말한다.

좋은 피드백:

```text
현재 프롬프트에는 목표와 출력 형식은 있습니다.
하지만 화면 크기 기준이 없어 김아이가 데스크톱 기준으로만 판단할 가능성이 큽니다.
다음 시도에서는 375px과 1024px에서의 배치 조건을 추가해보세요.
```

나쁜 피드백:

```text
프롬프트가 부족합니다.
더 잘 써보세요.
```

## Act 3: 김아이의 작업대와 회사 규칙판

### Objective

Act 3의 목적은 수강생이 좋은 프롬프트만으로는 충분하지 않다는 점을 확인하게 만드는 것이다.

Act 2에서 수강생은 좋은 업무 지시를 작성했다. Act 3에서는 같은 지시라도 김아이의 작업대 위에 어떤 자료가 올라가느냐에 따라 결과물이 개선되거나 오염되는 것을 확인한다.

Act 3은 단순한 자료 분류 게임이 아니다. 수강생이 직접 컨텍스트 자료를 고르고, 그 선택된 자료로 김아이가 실제 HTML/CSS 결과물을 만들고, 그 결과를 렌더링해서 본다.

### Core Message

```text
좋은 프롬프트도 작업대 위 자료가 섞이면 결과가 흔들립니다.
```

### Teaching Flow

Act 3의 강의 흐름은 다음 순서를 따른다.

1. `Act 2 결과 회수`: 좋은 프롬프트를 써도 매번 모든 정보를 다시 쓰는 것은 비효율적이라고 짚는다.
2. `작업대 비유`: 컨텍스트는 김아이가 지금 작업할 때 눈앞에 올려둔 자료라고 설명한다.
3. `오염 문제 제시`: 오래된 제목, 폐기된 디자인, 미확정 정책이 섞이면 김아이가 그것을 현재 조건으로 착각할 수 있음을 보여 준다.
4. `컨텍스트 선택 실습`: 수강생이 자료 후보 중 김아이에게 줄 자료를 고른다.
5. `결과 생성`: 선택된 자료를 바탕으로 김아이가 카드 컴포넌트를 다시 만든다.
6. `렌더링 확인`: 생성된 HTML/CSS 결과를 화면에서 확인한다.
7. `오염 분석`: 어떤 자료가 반영됐고, 어떤 자료가 결과를 흐렸는지 본다.
8. `재시도`: 자료를 줄이거나 바꿔 다시 생성한다.
9. `회사 규칙판 연결`: 매번 필요한 규칙은 프롬프트나 일회성 컨텍스트가 아니라 회사 규칙판으로 분리한다고 연결한다.

### Slide Role In Act 3

Act 3는 7-9장 정도의 슬라이드로 구성한다.

권장 슬라이드 흐름:

1. `Bridge Slide`: 매번 모든 맥락을 다시 쓰는 것은 비효율적이다.
2. `Concept Slide`: 컨텍스트는 김아이의 작업대 위 자료다.
3. `Contrast Slide`: 좋은 자료는 결과를 돕고, 오래된 자료는 결과를 오염시킨다.
4. `Practice Slide`: 김아이에게 줄 자료를 직접 고른다.
5. `System Slide`: 선택된 자료가 프롬프트와 함께 생성 요청으로 들어간다.
6. `Practice/System Slide`: 렌더링 결과와 컨텍스트 영향 분석을 본다.
7. `Contrast Slide`: Attempt 1과 Attempt 2의 자료 선택과 결과를 비교한다.
8. `Concept Slide`: 매번 필요한 규칙은 회사 규칙판으로 분리한다.
9. `Bridge Slide`: 반복되는 절차는 매번 다시 설명하지 말고 매뉴얼로 빼야 한다.

### Practice Name

```text
실습 3: 김아이의 작업대를 세팅합니다
```

### Practice Goal

수강생이 작업 목표에 맞는 컨텍스트 자료를 고르고, 선택한 자료가 실제 산출물을 어떻게 바꾸는지 확인한다.

Act 3는 `정보를 어디에 둘지`만 묻지 않는다. 선택한 자료로 실제 결과물을 만들고, 그 결과가 좋아졌는지 오염됐는지 검증한다.

### Learner Task

기본 과제:

```text
Act 2에서 만든 반응형 카드 컴포넌트를
"AI 하네스 워크숍 신청 카드"로 개선하세요.
```

수강생은 자료 후보 중 김아이에게 줄 컨텍스트를 선택한다. 선택지는 딱 떨어지는 정답/오답으로 만들지 않는다. 모든 자료는 그럴듯하지만, 지금 목표에 반드시 필요한 자료와 과한 자료, 오래된 자료, 아직 쓰면 안 되는 자료가 섞여 있다.

### Context Candidate Pool

Act 3의 자료 후보는 15-20개 정도로 구성한다.

기본 후보:

```text
[ ] 현재 워크숍 대상: AI 도구를 쓰기 시작한 실무자
[ ] 카드에 반드시 들어갈 문구: "4시간 실습형 워크숍"
[ ] 모바일에서는 CTA 버튼이 첫 화면 안에 보여야 한다
[ ] 브랜드 컬러: #2563eb, #111827
[ ] 기존 버튼 스타일은 유지한다
[ ] 지난달 폐기한 보라색 그라디언트 시안
[ ] 초기 기획안의 제목: "AI 자동화 마스터 클래스"
[ ] 강의 최종 제목: "AI 에이전트 하네스 엔지니어링"
[ ] 참고용 경쟁사 카드 5개
[ ] 강사가 개인적으로 좋아하는 글래스모피즘 스타일
[ ] 예전 수강생 피드백: "용어가 너무 어렵다"
[ ] 다음 섹션에서 다룰 Hook 상세 설명
[ ] 회사 소개 페이지의 mission 문구
[ ] 카드에는 가격 정보를 넣지 않는다
[ ] 375px에서 제목은 두 줄 이하로 유지한다
[ ] 데스크톱에서는 3열 카드 레이아웃의 일부로 들어간다
[ ] 내부 회의에서 나온 미확정 할인 정책
[ ] 접근성 기준: 버튼 focus 상태 필요
```

선택지 설계 의도:

- 필수 컨텍스트: 현재 대상, 최종 제목, 필수 문구, 모바일/데스크톱 기준, 접근성 기준, 가격 제외 규칙
- 유용한 컨텍스트: 브랜드 컬러, 기존 버튼 스타일, 수강생 피드백
- 조건부 컨텍스트: 경쟁사 카드, 디자인 선호, 회사 mission
- 오염 컨텍스트: 폐기된 시안, 초기 제목, 미확정 할인 정책
- 범위 밖 컨텍스트: Hook 상세 설명처럼 지금 카드 개선에는 필요 없는 뒤 섹션 내용

### Practice Loop

Act 3는 최소 2회, 가능하면 3회 시도를 전제로 한다.

필수 루프:

1. `Attempt 1`: 수강생이 자료 후보 중 김아이에게 줄 컨텍스트를 선택한다.
2. `Generate`: 선택한 자료와 Act 2의 프롬프트를 함께 사용해 카드 컴포넌트를 생성한다.
3. `Render`: 생성된 HTML/CSS 결과를 미리보기로 렌더링한다.
4. `Analyze`: 반영된 자료, 무시된 자료, 충돌한 자료, 오염시킨 자료를 표시한다.
5. `Feedback`: 어떤 자료를 빼거나 추가해야 하는지 힌트를 준다.
6. `Attempt 2`: 컨텍스트 선택을 조정하고 다시 생성한다.
7. `Compare`: Attempt 1과 Attempt 2의 결과, 점수, 렌더링 차이를 비교한다.
8. `Save`: 가장 좋은 컨텍스트 세트와 결과물을 저장한다.

Act 3의 반복은 프롬프트 문장 수정이 아니라 컨텍스트 자료 선택 수정에 초점을 둔다.

### Result Screen

```text

- HTML/CSS 코드 요약
- 이전 시도 대비 점수 변화

- 반영된 자료
- 무시된 자료
- 충돌한 자료
- 결과를 오염시킨 자료

검증 로그
- final-title-check: pass/fail

- responsive-context-check: pass/fail
- style-consistency-check: pass/fail
- accessibility-context-check: pass/fail
- scope-noise-check: pass/fail

비교
- Attempt 1: 선택 자료 수, 점수, 주요 오염 자료
- Attempt 2: 선택 자료 수, 점수, 개선된 점
```

결과 화면은 `자료를 많이 넣었는가`가 아니라 `현재 목표에 맞는 자료를 정확히 골랐는가`를 보여 줘야 한다.

### Scoring In Act 3

Act 3의 점수는 자료 선택 점수와 최종 산출물 검증 점수를 함께 사용한다.

기본 점수 축:

```text
필수 컨텍스트 포함: +점수
유용한 컨텍스트 포함: +소점수
조건부 컨텍스트 과다 포함: -점수

```

검증 예시:

```text
final-title-check

stale-context-check

responsive-context-check
```

### Company Rule Board Connection

Act 3 후반에는 `회사 규칙판` 개념을 도입한다. 이때 바로 CLAUDE.md 파일 구조를 깊게 설명하지 않는다.

```text
필요할 때만 열어볼 자료인가?
오래되어 제거해야 하는가?
```

```text
매번 반복되는 회사 규칙은 프롬프트에 계속 붙이지 않습니다.
항상 유지되는 규칙판으로 분리합니다.
```

### Feedback Tone

피드백은 `자료를 많이 골랐다/적게 골랐다`가 아니라 `현재 목표에 맞는 작업대인가`로 말한다.

좋은 피드백:

```text
최종 강의 제목과 모바일 조건은 잘 반영됐습니다.
하지만 폐기된 보라색 그라디언트 시안이 같이 들어가 김아이가 현재 디자인 방향으로 오해했습니다.
다음 시도에서는 오래된 시안과 미확정 할인 정책을 빼고, 버튼 focus 기준을 추가해 보세요.
```

나쁜 피드백:

```text
컨텍스트 선택이 틀렸습니다.
자료를 줄이세요.
```

### What The Instructor Says From The Slide

슬라이드에는 긴 대본을 넣지 않는다. 발표자는 다음 흐름만 보고 말한다.

```text
헤드라인:
좋은 프롬프트도 작업대 위 자료가 섞이면 결과가 흔들립니다.

3앵커:
1. 지금 필요한 자료
2. 오래된 오염 자료
3. 항상 남길 회사 규칙

브릿지:
자료를 잘 골라도, 반복되는 절차를 매번 다시 설명하면 또 흔들립니다.
```

### Bridge To Act 4

Act 3의 마지막 질문:

```text
자료를 잘 골랐습니다. 그런데 이 작업 절차를 매번 다시 설명해야 할까요?
```

이 질문이 Act 4로 넘어가는 문이다. Act 4에서는 반복되는 절차를 Skill, Superpowers 같은 업무 매뉴얼로 분리한다.

### Acceptance Criteria

Act 3 설계는 다음 조건을 만족해야 한다.

- 수강생이 컨텍스트 자료 후보를 직접 선택한다.
- 선택지가 명백한 정답/오답이 아니라 실무적으로 애매해야 한다.
- 선택한 자료로 실제 HTML/CSS 결과물이 생성된다.
- 결과물이 렌더링 미리보기로 표시된다.
- 반영된 자료, 무시된 자료, 충돌한 자료, 오염시킨 자료가 표시된다.
- 최소 2회 이상 컨텍스트 선택을 바꿔 재시도할 수 있다.
- Attempt 간 결과와 점수 차이를 비교할 수 있다.
- 회사 규칙판과 CLAUDE.md의 필요성이 자연스럽게 열린다.
- Act 4의 Skill/Superpowers 필요성이 자연스럽게 열린다.

### Out Of Scope For Act 3

Act 3에서는 다음을 깊게 다루지 않는다.

- Skill 작성법
- Superpowers 전체 워크플로우
- Agent/Subagent 역할 설계
- MCP 서버와 Tool 권한 설계
- Hook 설정
- Evaluation harness 전체 구조
- CLAUDE.md의 모든 로딩 경로와 세부 설정

이 항목들은 뒤 Act에서 각각 다룬다. Act 3은 오직 `컨텍스트 선택이 실제 결과물을 바꾼다`를 렌더링과 오염 분석으로 확인하는 데 집중한다.

## Act 4: 반복 업무는 Skill로 꺼내 둡니다

### Objective

Act 4의 목적은 수강생이 반복되는 작업 절차를 매번 프롬프트에 다시 쓰는 대신, 재사용 가능한 Skill로 분리하는 감각을 갖게 만드는 것이다.

Act 3에서 수강생은 좋은 자료 선택이 결과물을 바꾼다는 것을 확인했다. Act 4에서는 AI가 성급하게 답하거나 바로 구현하지 못하도록, 작업 전에 요구사항을 좁히는 `미니 브레인스토밍 Skill`을 직접 만든다.

Act 4는 Skill을 설명으로만 다루지 않는다. 수강생이 자기 AI 에이전트 프로그램에서 Skill 문서를 작성하고, 그 내용을 실습 페이지에 붙여넣어 하네스 검증과 점수를 받는다.

### Core Message

```text
Skill은 답변 템플릿이 아니라, AI가 일하는 방식을 강제하는 절차입니다.
```

### Teaching Flow

Act 4의 강의 흐름은 다음 순서를 따른다.

1. `Act 3 결과 회수`: 자료를 잘 골라도 반복되는 절차를 매번 다시 설명하면 흔들린다고 짚는다.
2. `성급한 AI 문제 제시`: 사용자가 열린 요청을 했는데 AI가 바로 구현안이나 아이디어를 쏟아내는 실패를 보여 준다.
3. `Skill 비유`: Skill은 AI에게 매번 말하던 반복 절차를 파일로 만든 업무 매뉴얼이라고 설명한다.
4. `Superpowers 연결`: brainstorming Skill은 구현 전에 목표와 범위를 좁히게 만드는 작업 규율이라고 소개한다.
5. `Skill 기본 구조 소개`: name, description, 사용 조건, 절차, 승인 게이트, 출력 형식을 보여 준다.
6. `실습 안내`: 수강생이 자기 AI 에이전트 프로그램에서 미니 브레인스토밍 Skill을 작성한다.
7. `붙여넣기 검증`: 작성한 Skill 내용을 실습 페이지에 붙여넣는다.
8. `하네스 채점`: 구조 검사, 질문 방식, 범위 분리, 승인 전 구현 금지, 스펙 기록을 점수화한다.
9. `Act 5 연결`: 좋은 Skill이 있어도 조사, 구현, 리뷰를 한 명의 김아이가 모두 보면 판단이 섞인다고 연결한다.

### Slide Role In Act 4

Act 4는 7-9장 정도의 슬라이드로 구성한다.

권장 슬라이드 흐름:

1. `Bridge Slide`: 반복되는 절차를 매번 다시 설명하면 또 흔들린다.
2. `Contrast Slide`: 바로 구현하는 김아이와 먼저 질문하는 김아이를 비교한다.
3. `Concept Slide`: Skill은 반복 업무를 파일로 만든 매뉴얼이다.
4. `System Slide`: Skill은 호출 조건, 절차, 승인 게이트, 출력 형식으로 구성된다.
5. `Practice Slide`: 미니 브레인스토밍 Skill을 직접 작성한다.
6. `System Slide`: 붙여넣은 Skill을 하네스가 구조 검사와 LLM judge로 평가한다.
7. `Practice/System Slide`: 점수를 보고 Skill을 고쳐 다시 제출한다.
8. `Concept Slide`: Superpowers는 이런 Skill들을 작업 루프에 배치한 규율 패키지다.
9. `Bridge Slide`: 매뉴얼이 있어도 역할이 섞이면 판단이 흔들린다.

### Practice Name

```text
실습 4: 미니 브레인스토밍 Skill을 만듭니다
```

### Practice Goal

수강생이 열린 요청을 받았을 때 AI가 바로 구현하지 않고, 먼저 목적과 범위와 접근안을 좁히도록 만드는 Skill을 작성한다.

이 실습의 산출물은 단순 답변이 아니라 수강생이 자기 AI 에이전트 프로그램에 붙여 넣을 수 있는 Skill 문서다.

### Learner Task

과제:

```text
당신의 AI 에이전트가 새 기능, 강의 구성, 실습 아이디어, 화면 개선처럼 범위가 열린 요청을 받았을 때 사용할 미니 브레인스토밍 Skill을 만드세요.
이 Skill은 언제 사용되는지, 어떤 질문을 어떤 순서로 해야 하는지, 언제 구현을 멈춰야 하는지, 결정 내용을 어떤 형식으로 남길지 포함해야 합니다.
```

수강생 행동:

1. 강의에서 Skill 구조와 Superpowers brainstorming의 목적을 본다.
2. 자기 AI 에이전트 프로그램에 미니 브레인스토밍 Skill을 작성한다.
3. 작성한 Skill 내용을 실습 페이지에 붙여넣는다.
4. 하네스가 Skill 구조와 품질을 검증한다.
5. 점수, 누락 항목, 개선안을 보고 수정한다.
6. 다시 제출하고 점수 변화를 확인한다.

### Bad Behavior Example

Act 4 시작에는 Skill이 없을 때의 나쁜 김아이 행동을 보여 준다.

```text
사용자: 강의 실습 좀 재밌게 바꿔줘.
김아이: 바로 20개의 실습 아이디어와 구현 계획을 한 번에 쏟아냄.
문제: 목표, 청중, 시간, 구현 범위, 승인 여부가 정리되지 않았다.
```

이 실패를 본 뒤, 수강생은 AI가 성급하게 움직이지 못하게 만드는 Skill을 작성한다.

### Expected Skill Shape

수강생 제출물은 특정 도구에 완전히 종속되지 않아야 한다. 다만 기본적으로 다음 구조를 권장한다.

```md
---
name: mini-brainstorming
description: 사용자가 새 기능, 강의 구성, 실습 아이디어, 화면 개선처럼 범위가 열린 작업을 요청할 때 사용한다.
---

# Mini Brainstorming

## When to Use

새 기능을 만들거나 기존 작업을 크게 바꾸기 전에 사용한다.
요구가 모호하거나, 구현 전에 방향을 좁혀야 할 때 사용한다.

## Steps

1. 현재 목표를 한 문장으로 다시 말한다.
2. 사용자가 원하는 결과물과 원하지 않는 범위를 구분한다.
3. 한 번에 하나의 질문만 한다.
4. 답변을 바탕으로 2~3개의 접근안을 제안한다.
5. 장단점을 비교하고 추천안을 고른다.
6. 사용자가 승인하기 전에는 구현하지 않는다.
7. 결정된 내용을 짧은 스펙으로 남긴다.

## Output Format

- 현재 목표
- 확인한 제약
- 열린 질문
- 접근안
- 추천안
- 다음 결정
```

이 예시는 원본 Superpowers brainstorming을 복제하는 모범답안이 아니다. 수강생은 자기 업무와 자기 AI 도구에 맞는 미니 버전을 작성한다.

### Secondary Example

자료조사 Skill은 Act 4의 메인 실습이 아니라 보조 예시로만 다룬다.

```text
자료조사 Skill은 최신성, 공식 출처, 불확실성 보고를 반복 절차로 만든 예시입니다.
오늘 직접 만드는 것은 작업 전에 요구사항을 좁히는 미니 브레인스토밍 Skill입니다.
```

### Practice Loop

Act 4는 최소 2회, 가능하면 3회 제출을 전제로 한다.

필수 루프:

1. `Attempt 1`: 처음 만든 Skill을 제출한다.
2. `Structure Check`: frontmatter, name, description, steps, output format을 검사한다.
3. `Behavior Check`: 한 번에 하나씩 질문하는지, 범위를 분리하는지, 승인 전 구현을 막는지 검사한다.
4. `Feedback`: 누락 항목과 개선 제안을 확인한다.
5. `Attempt 2`: description과 steps를 보강해 다시 제출한다.
6. `Compare`: 이전 점수와 현재 점수, 개선된 항목을 비교한다.
7. `Attempt 3`: 승인 게이트와 스펙 기록 형식을 보강한다.
8. `Save`: 최종 Skill을 저장한다.

Act 4의 반복은 프롬프트 수정이 아니라 Skill 문서의 호출 조건, 질문 절차, 승인 게이트, 출력 형식을 개선하는 데 초점을 둔다.

### Mechanical Scoring Logic

1차 채점은 기계적으로 한다.

점수 기준:

```text
호출 조건이 명확한가: 20점
한 번에 하나씩 질문하게 되어 있는가: 15점
원하는 것과 원하지 않는 범위를 구분하는가: 15점
2~3개 접근안을 제안하고 trade-off를 비교하는가: 15점
사용자가 승인하기 전 구현하지 않도록 되어 있는가: 15점
결정 내용을 스펙으로 남기게 하는가: 10점
출력 형식이 있는가: 10점
```

감점 후보:

```text
description이 "아이디어가 필요할 때 사용"처럼 너무 넓다.
질문을 한 번에 여러 개 던지게 되어 있다.
접근안 비교 없이 바로 추천안만 말한다.
사용자 승인 전 구현 금지 규칙이 없다.
결정과 미결정을 구분하지 않는다.
스펙이나 요약을 남기는 출력 형식이 없다.
```

### LLM Judge Role

LLM judge는 2차 평가만 맡는다. 총점을 단독으로 결정하지 않는다.

LLM judge 평가 항목:

```text
- AI가 성급하게 구현하지 못하게 막는가
- 질문이 너무 많아 사용자를 지치게 하지 않는가
- 결정과 미결정을 구분하는가
- 접근안의 trade-off가 실제 판단에 도움이 되는가
- 실제 업무 전에 쓸 수 있는가
```

LLM judge 결과 예:

```text
llm-judge: good with gaps
reason: 요구사항을 좁히는 절차는 있지만, 사용자 승인 전 구현 금지 규칙이 명확하지 않습니다.
```

### Score Titles

Act 4는 최종 점수에 따라 다음 타이틀을 부여한다.

```text
90~100점: 요구사항 정리 전문가
75~89점: 쓸 수 있는 브레인스토밍 Skill
55~74점: 질문은 하지만 흔들리는 Skill
35~54점: 아이디어 템플릿 수준
0~34점: 바로 구현해버리는 김아이
```

### Superpowers Connection

Act 4 후반에는 Superpowers를 도구명이 아니라 작업 규율 패키지로 소개한다.

연결 문장:

```text
우리가 방금 만든 것은 미니 brainstorming Skill입니다.
Superpowers는 이런 Skill들을 브레인스토밍, 디버깅, 검증, 리뷰 같은 개발 루프에 배치한 작업 규율 패키지입니다.
```

Act 4에서 Superpowers의 모든 스킬을 설명하지 않는다. 핵심은 `Skill 하나를 만들 수 있다면, AI가 일하는 방식을 반복 가능한 절차로 바꿀 수 있다`는 감각이다.

### What The Instructor Says From The Slide

슬라이드에는 긴 대본을 넣지 않는다. 발표자는 다음 흐름만 보고 말한다.

```text
헤드라인:
Skill은 답변 템플릿이 아니라 AI가 일하는 방식을 강제하는 절차입니다.

3앵커:
1. 호출 조건
2. 질문 절차
3. 승인 게이트

브릿지:
좋은 Skill이 있어도, 김아이 혼자 조사와 구현과 리뷰를 모두 맡으면 판단이 섞입니다.
```

### Bridge To Act 5

Act 4의 마지막 질문:

```text
좋은 Skill이 있어도, 김아이 혼자 조사, 구현, 리뷰를 모두 맡으면 괜찮을까요?
```

이 질문이 Act 5로 넘어가는 문이다. Act 5에서는 Agent/Subagent, Tool, MCP, 권한 분리를 다룬다.

### Acceptance Criteria

Act 4 설계는 다음 조건을 만족해야 한다.

- 수강생이 직접 미니 브레인스토밍 Skill을 작성한다.
- 수강생이 자기 AI 에이전트 프로그램에서 작성한 내용을 붙여넣을 수 있다.
- 하네스가 Skill 구조와 행동 품질을 채점한다.
- 최소 2회 이상 Skill을 수정해 다시 제출할 수 있다.
- 기계 채점과 LLM judge 역할이 분리되어 보인다.
- 결과 화면에 누락 항목, 개선 제안, 점수 변화가 표시된다.
- 최종 Skill은 수강생이 실제 업무에서 복사해 사용할 수 있는 형태여야 한다.
- Superpowers는 Skill 묶음과 작업 규율 패키지로 자연스럽게 연결된다.
- 자료조사 Skill은 보조 예시로만 등장한다.
- Act 5의 Agent/Subagent 필요성이 자연스럽게 열린다.

### Out Of Scope For Act 4

Act 4에서는 다음을 깊게 다루지 않는다.

- Agent/Subagent 역할 설계
- MCP 서버와 Tool 권한 설계
- Hook 설정
- Evaluation harness 전체 구조
- Superpowers 전체 스킬 목록 암기
- 특정 AI 도구별 Skill 설치 경로 전체 설명
- 원본 Superpowers brainstorming Skill 복제

이 항목들은 뒤 Act에서 각각 다룬다. Act 4는 오직 `작업 전 요구사항을 좁히는 절차를 Skill로 분리하고 검증받는다`에 집중한다.

## Act 5: 김아이 팀에 Skill과 도구를 나눠 줍니다

### Objective

Act 5의 목적은 수강생이 AI 에이전트 팀을 단순히 여러 명으로 늘리는 것이 아니라, 역할, Skill, 컨텍스트, 도구 권한을 분리하는 구조로 이해하게 만드는 것이다.

Act 4에서 수강생은 미니 브레인스토밍 Skill을 만들었다. Act 5에서는 이 Skill을 아무 역할에게나 주지 않고, Coordinator에게 붙여서 작업 시작 전에 목표와 제약을 좁히게 한다. 다른 역할은 조사, 구현, 리뷰를 분리해 맡는다.

Act 5는 웹 채점 실습이 아니다. 수강생이 자기 로컬 AI 에이전트 환경에서 직접 역할 프롬프트와 Skill/Tool 배정을 적용하고 실행해 보는 로컬 실행형 실습이다.

### Core Message

```text
에이전트 팀은 AI를 여러 명 늘리는 것이 아니라, 역할과 Skill과 도구 권한을 분리하는 구조입니다.
```

### Teaching Flow

Act 5의 강의 흐름은 다음 순서를 따른다.

1. `Act 4 결과 회수`: 좋은 Skill이 있어도 한 명의 김아이가 조사, 구현, 리뷰를 모두 맡으면 판단이 섞인다고 짚는다.
2. `역할 분리 소개`: Coordinator, Researcher, Implementer, Reviewer 역할을 나눈다.
3. `Skill 배정 소개`: Act 4에서 만든 mini-brainstorming Skill은 Coordinator에게 붙인다.
4. `도구 권한 소개`: 각 역할에 필요한 도구만 준다.
5. `MCP 연결`: Tool은 실제 행동이고, MCP는 도구를 AI 앱에 연결하는 표준 통로라고 설명한다.
6. `로컬 실습 안내`: 강의 자료의 템플릿을 복사해 자기 로컬 AI 도구에 붙인다.
7. `실행`: Coordinator prompt로 역할 분리 작업을 실행한다.
8. `기록`: 결과 로그와 역할 섞임, 빠진 검증, 개선점을 기록한다.
9. `2차 시도`: 역할, Skill, 도구 권한을 바꿔 다시 실행한다.
10. `Act 6 연결`: 역할과 도구를 나눠도 완료를 말로만 믿으면 충분하지 않다고 연결한다.

### Slide Role In Act 5

Act 5는 7-9장 정도의 슬라이드로 구성한다.

권장 슬라이드 흐름:

1. `Bridge Slide`: 좋은 Skill이 있어도 한 명이 다 맡으면 판단이 섞인다.
2. `Concept Slide`: 에이전트 팀은 역할, Skill, 도구 권한의 조합이다.
3. `System Slide`: Coordinator, Researcher, Implementer, Reviewer의 책임을 분리한다.
4. `System Slide`: 각 역할에 어떤 Skill과 Tool을 줄지 보여 준다.
5. `Practice Slide`: 로컬 AI 도구에 역할 프롬프트를 붙여넣는다.
6. `Practice Slide`: Coordinator prompt를 실행한다.
7. `Reflection Slide`: 실행 기록 템플릿에 결과를 정리한다.
8. `Concept Slide`: MCP는 도구 연결의 표준 통로다.
9. `Bridge Slide`: 역할을 나눠도 최종 검증 게이트가 없으면 흔들린다.

### Practice Name

```text
실습 5: 로컬에서 김아이 팀을 실행합니다
```

### Practice Goal

수강생이 자기 컴퓨터의 AI 에이전트 도구에서 작은 에이전트 팀을 구성하고 실행한다.

이 실습은 점수화보다 체험과 기록이 중요하다. 웹 페이지에서 검증하지 않는다. 대신 강의 자료가 역할 프롬프트, Skill/Tool 배정 예시, 실행 체크리스트, 결과 기록 템플릿을 제공한다.

### Learner Task

과제:

```text
AI 하네스 워크숍 신청 카드 컴포넌트를 최종 개선하세요.
단, 한 명의 김아이에게 모두 맡기지 말고 역할을 나눠 실행하세요.
```

수강생 행동:

1. 강의 페이지에서 역할별 프롬프트 템플릿을 복사한다.
2. 자기 로컬 AI 에이전트 도구에 Coordinator, Researcher, Implementer, Reviewer 역할을 등록하거나 역할 프롬프트로 붙인다.
3. Act 4에서 만든 mini-brainstorming Skill을 Coordinator에게 붙인다.
4. 역할별 Skill과 Tool 권한을 제한한다.
5. Coordinator prompt로 작업을 시작한다.
6. 역할별 출력과 최종 결과를 확인한다.
7. 실행 기록 템플릿에 결과를 정리한다.
8. 2차 시도에서 역할, Skill, 도구 권한을 바꿔본다.

### Team Roles

기본 팀은 네 역할로 제한한다.

```text

- 목표와 제약을 정리한다.
- Act 4의 mini-brainstorming Skill을 먼저 적용한다.
- 역할별 결과를 통합한다.
- 직접 구현하지 않는다.

Researcher
- 요구사항과 근거를 확인한다.
- 구현하지 않는다.
- 불확실한 점과 구현자에게 넘길 정보를 정리한다.

Implementer
- HTML/CSS 결과물을 만든다.
- 출처 판단이나 최종 품질 판정은 하지 않는다.
- 확인이 필요한 부분을 남긴다.

Reviewer
- 결과물의 요구사항 누락, 반응형 실패, 접근성 문제를 찾는다.
- 직접 수정하지 않는다.
- 발견한 문제와 남은 위험을 보고한다.
```

```text
Implementer

- Tool: Read, Write/Edit, Preview
- 권한: HTML/CSS 생성, 최종 품질 판정 금지

Reviewer
- Skill: review-checklist 또는 검토 절차 프롬프트
- Tool: Read, Preview
- 권한: 결함 발견, 직접 수정 금지
```

## Act 6: 완료 전에 자동 검문소를 세웁니다

### Objective

Act 6의 목적은 수강생이 AI의 `완료했습니다`라는 말을 그대로 믿지 않고, 완료 직전에 자동으로 확인하는 검문소를 설계하게 만드는 것이다.

Act 5에서 수강생은 역할, Skill, Tool 권한을 나눠 로컬 에이전트 팀을 실행했다. Act 6에서는 역할을 나눠도 최종 완료 판정이 없으면 다시 흔들린다는 점을 확인하고, 범용 Stop hook 기반 완료 전 검문소를 만든다.

Act 6은 일반인 대상 강의이므로 처음부터 Hook JSON이나 스크립트를 설명하지 않는다. 먼저 체크리스트형 검문소를 설계하고, 100점을 맞춘 뒤 실제 Stop hook을 만들 수 있는 프롬프트와 예제 구조를 제공한다.

### Core Message

```text
완료는 말이 아니라, 자동 검문소를 통과한 증거로 판단해야 합니다.
```

### Teaching Flow

Act 6의 강의 흐름은 다음 순서를 따른다.

### Practice Goal

수강생이 체크리스트형 실습 화면에서 완료 전 자동 검문소를 구성한다. 100점을 맞추면 범용 Stop hook을 만들 수 있는 프롬프트와 예제 파일 구성이 열린다.

실습의 목적은 Hook 설정을 외우는 것이 아니라 다음 구조를 이해하는 것이다.

```text
새 요청이 들어오면 status = processing
Claude가 멈추려 하면 Stop hook이 완료 증거를 검사
빠진 작업이 있으면 계속 작업하도록 block
완료 증거가 충분하면 status = complete
status가 complete이면 Stop hook은 통과
```

### Confirmed Hook Design Decision

Act 6에서 사용할 Hook 구조는 다음 방식으로 확정한다.

```text
1. UserPromptSubmit hook에서 새 사용자 요청이 들어오면 status를 processing으로 바꾼다.
2. 작업 중에는 status 파일을 기준으로 현재 작업 상태를 추적한다.
3. Stop hook은 Claude가 멈추려는 순간 status와 완료 증거를 확인한다.
4. status가 processing이고 완료 증거가 부족하면 block으로 멈춤을 막는다.
5. 더 진행할 작업이 없고 완료 증거가 충분하면 status를 complete로 바꾼다.
6. status가 complete이면 Stop hook은 allow로 통과한다.
7. stop_hook_active와 attempts를 함께 사용해 Stop hook 재진입과 무한 반복을 막는다.
```

이 결정의 핵심은 `계속 작업하게 만드는 Hook`이 아니라 `작업이 덜 끝났을 때만 계속하게 하고, 끝났을 때는 반드시 멈출 수 있는 Hook`을 만드는 것이다. 일반인 대상 강의에서는 이 구조를 다음 비유로 설명한다.

```text
UserPromptSubmit = 새 업무 접수대
status file = 업무 진행표
Stop hook = 퇴근 전 검문소
stop_hook_active = 이미 검문 중인지 확인하는 표식
attempts = 같은 검문소를 몇 번 통과하려 했는지 세는 횟수
complete = 오늘 업무를 닫아도 된다는 승인 상태
```

발표에서는 먼저 이 비유를 보여 주고, 이후에 `.agent-status.json`, `UserPromptSubmit`, `Stop`, `stop_hook_active`라는 실제 이름으로 연결한다.

### Example Stop Hook Script

이 예시는 교육용 최소 버전이다. 실제 프로젝트에서는 프로젝트별 테스트, 빌드, 리뷰 명령을 추가할 수 있다.

```js
#!/usr/bin/env node
const fs = require("node:fs");
const path = require("node:path");

const statusPath = path.join(process.cwd(), ".agent-status.json");

function readStdin() {
  return new Promise((resolve) => {
    let data = "";
    process.stdin.setEncoding("utf8");
    process.stdin.on("data", (chunk) => (data += chunk));
    process.stdin.on("end", () => resolve(data));
  });
}

function output(value) {
  process.stdout.write(`${JSON.stringify(value)}\n`);
}

(async () => {
  const inputText = await readStdin();
  const input = inputText ? JSON.parse(inputText) : {};

  if (!fs.existsSync(statusPath)) {
    output({
      decision: "block",
      reason: "작업 상태 파일이 없습니다. 먼저 작업 상태를 processing으로 시작하세요."
    });
    return;
  }

  const status = JSON.parse(fs.readFileSync(statusPath, "utf8"));

  if (status.status === "complete") {
    output({ decision: "allow" });
    return;
  }

  if (input.stop_hook_active && status.attempts >= 1) {
    status.status = "complete";
    status.completedAt = new Date().toISOString();
    fs.writeFileSync(statusPath, `${JSON.stringify(status, null, 2)}\n`);
    output({ decision: "allow" });
    return;
  }

  const lastMessage = input.last_assistant_message || "";
  const hasEvidence =
    /검증|테스트|확인|실행|통과|PASS|완료/.test(lastMessage) &&
    /남은 위험|미해결|확인 못|제한/.test(lastMessage);

  if (!hasEvidence) {
    status.attempts = (status.attempts || 0) + 1;
    fs.writeFileSync(statusPath, `${JSON.stringify(status, null, 2)}\n`);

    output({
      decision: "block",
      reason:
        "완료 전 검문소가 멈춤을 막았습니다. 최종 답변에 실행한 검증, 통과/실패 결과, 남은 위험 또는 확인하지 못한 범위를 포함하세요."
    });
    return;
  }

  status.status = "complete";
  status.completedAt = new Date().toISOString();
  fs.writeFileSync(statusPath, `${JSON.stringify(status, null, 2)}\n`);
  output({ decision: "allow" });
})();
```

### Important Safety Notes

- 이 예시는 범용 교육용이다. 실제 프로젝트에서는 검증 명령과 완료 기준을 프로젝트에 맞게 바꿔야 한다.
- Stop hook이 항상 block하면 무한 반복이 생긴다.
- `stop_hook_active`와 `attempts`를 함께 사용해 반복을 제한한다.
- `complete` 상태일 때는 Stop hook이 통과해야 한다.
- Claude Code 공식 Hook 동작에서 Stop hook은 Claude가 멈추려 할 때 실행되며, `decision: "block"`은 멈춤을 막고 계속 작업하게 한다.
