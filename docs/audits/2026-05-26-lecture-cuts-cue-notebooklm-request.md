상세 발표 스크립트 앞에서 우리는 좋은 요청을 위해 목표, 검증, 보고 형식을 쪼개는 방법을 알아봤습니다. 이걸 보시고 나면 실무자분들이 종종 이런 질문을 하십니다. &quot;오타 하나 고치고 간단한 함수 하나 만드는데도 매번 이렇게 장황하게 지시를 내려야 하나요?&quot; 정답은 '아닙니다'입니다. 많은 분들이 프롬프트를 꼼꼼하게 쓰는 법을 배우고 나면, 모든 요청에 온갖 예시와 계획, 검증 절차를 꽉꽉 눌러 담으려는 함정에 빠집니다. 그러면 AI는 너무 많은 정보에 짓눌려서 오히려 간단한 일을 복잡하게 망쳐버리기도 합니다. 우리의 목표는 무조건 길고 무거운 요청서를 작성하는 게 아닙니다. 작업의 성격과 위험도에 맞춰서 안전장치의 무게를 조절하는 겁니다. 가볍게 동네 마트에 가는데 F1 경주용 6점식 안전벨트를 맬 필요는 없는 것처럼요. 그래서 지시의 강도를 조절해야 합니다. 단순한 변수명 변경이나 가벼운 번역 같은 간단한 일은 목표만 선명하게 던져주면 충분합니다. 반면에, 출력되는 텍스트의 구조나 디자인이 아주 중요한 작업이라면 말로 길게 설명하기보다 잘 만들어진 예시 하나를 툭 붙여주는 게 훨씬 정확합니다. 그런데 만약 실제 서버의 데이터를 건드리거나 결제 로직을 수정하는 아주 위험한 일이라면 어떨까요? 이때는 절대 AI의 판단에만 맡겨두면 안 됩니다. 반드시 통과해야 할 검증 목록과, 작업이 성공했다는 것을 어떻게 증명할지 명확한 보고 기준을 단단하게 채워 넣어야 합니다. 정보가 부족하다면 무작정 코드를 짜라고 시키기 전에 질문이나 조사를 먼저 하라고 요청 강도를 낮춰야 하고요. 결국 좋은 하네스 설계란, 이 작업이 얼마나 위험한지 파악하고 딱 그에 맞는 강도의 난간만 세워주는 감각을 의미합니다. 자, 여기까지가 한 번의 대화에서 지시를 내리는 프롬프트 단계의 이야기였습니다. 그런데 아무리 강도를 잘 조절해도, &quot;테스트를 꼭 돌려라&quot;, &quot;이 파일은 건드리지 마라&quot; 같은 말을 매번 반복해서 치고 있다면 어떨까요? 이제 매번 지시하는 단계를 넘어서, AI가 우리 프로젝트의 규칙을 항상 '기억'하게 만드는 다음 레이어로 넘어가 보겠습니다.

상세 원고 요약용 원문:
상세 발표 스크립트 이제 한 번의 지시인 프롬프트를 넘어서, AI의 '기억'을 다루는 메모리 레이어(Memory Layer)로 올라가 보겠습니다. AI와 일하다 보면 은근히 답답할 때가 있습니다. 매번 새 대화를 열 때마다 &quot;우리 프로젝트는 npm 말고 pnpm을 써라&quot;, &quot;제발 파일 고치기 전에 원본부터 읽어라&quot;, &quot;테스트 실패하면 마음대로 스킵하지 마라&quot; 같은 잔소리를 똑같이 반복해야 하거든요. 이런 반복적인 피드백을 매번 프롬프트에 길게 적어줄 수는 없습니다. 그래서 등장한 것이 바로 CLAUDE.md입니다. 쉽게 말해, AI가 작업 내내 모니터 옆에 항상 붙여두고 보는 '포스트잇'이자 '항상 켜진 기억'입니다. 프로젝트의 핵심 규칙이나 명령을 여기에 한 번만 적어두면, AI가 알아서 매 세션마다 이 규칙을 기본으로 탑재하고 일하게 됩니다. 그런데 여기서 실무자들이 흔히 하는 큰 실수가 있습니다. &quot;아, 그럼 우리 팀의 길고 복잡한 API 문서나 업무 매뉴얼도 다 여기에 텍스트로 때려 넣으면 알아서 똑똑해지겠네?&quot;라고 생각하시는 겁니다. 절대 안 됩니다. CLAUDE.md는 길게 쓸수록 오히려 독이 됩니다. 항상 켜져 있는 기억이기 때문에, 내용이 방대해지면 AI는 정작 가장 중요한 핵심 규칙을 놓쳐버리게 됩니다. 불필요한 정보가 매 작업마다 계속 따라다니니 컨텍스트 비용만 커지고, 결국 지시들끼리 충돌하는 일도 생기죠. 긴 API 문서나 두꺼운 업무 매뉴얼은 잠시 후 배울 '스킬(Skill)'이라는 별도의 도구 상자에 참고 자료로 분리해서, 필요할 때만 열어보게 만들어야 합니다. 그럼 이 파일에는 정확히 무엇만 남겨야 할까요? 아주 명확한 판단 기준이 하나 있습니다. 스스로에게 이렇게 질문해 보시는 겁니다. &quot;이 내용, 앞으로 AI한테 시킬 20번의 작업에서 한 번도 빠짐없이 항상 알아야 하는 규칙인가?&quot; 만약 그렇다면, CLAUDE.md에 들어갈 자격이 있습니다. 반대로 &quot;가끔 특정 작업을 할 때만 필요한가?&quot; 그렇다면 스킬(Skill)로 빼야 합니다. &quot;이건 말로 할 게 아니라 무조건 시스템적으로 강제 실행되게 막아야 하는 일인가?&quot; 그렇다면 나중에 배울 자동화 장치인 훅(Hook)으로 올려야 합니다. 결국 CLAUDE.md는 모든 지식을 담아두는 거대한 백과사전이 아니라, 우리 프로젝트가 흔들리지 않게 잡아주는 최소한의 '헌법'이 되어야 합니다. 그렇다면 이 중요한 프로젝트의 기억들은 구체적으로 어떤 우선순위와 계층으로 나뉘어서 작동하게 될까요? 다음 화면에서 그 구조를 함께 살펴보겠습니다.
