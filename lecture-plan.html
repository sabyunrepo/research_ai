AI 에이전트 하네스 엔지니어링 실전 강의

`CLAUDE.md`, Skills, Superpowers, Subagents, Hooks를 하나의 작업 시스템으로 엮어 “AI에게 잘 말하는 법”이 아니라 “AI가 안정적으로 일하게 만드는 구조”를 가르치는 강의안입니다. 실제 발표 순서는 lecture-cuts/assets/slides.js를 기준으로 한다.

대상: AI 코딩 도구 입문-중급 개발자 형식: 개념 45% · 해부 20% · 실습 35% 산출물: 프로젝트용 CLAUDE.md + Skill + Hook 워크플로우

강의 목표

1. 개념 정리

`CLAUDE.md`, Skills, Superpowers, Subagents, Hooks가 각각 어떤 레이어인지 구분합니다.

2. 구조 해부

Superpowers를 “스킬 묶음”이자 “하네스 엔지니어링 사례”로 해부합니다.

3. 실습 완성

프로젝트 규칙, 스킬, 훅을 조합해 작은 자동화 워크플로우를 직접 만듭니다.

핵심 프레임

요소 한 줄 정의 강의에서의 설명 포인트 실습 예
CLAUDE.md 매 세션 로드되는 프로젝트 운영 규칙 항상 알아야 하는 규칙만 넣고, 200줄 이하로 유지하는 원칙 빌드/테스트 명령, 브랜치 규칙, 검증 체크리스트 작성
Skills 필요할 때 로드되는 작업 매뉴얼 `description`이 자동 호출 품질을 결정하고, 긴 자료는 supporting files로 분리 `issue-triage` 또는 `verify-change` 스킬 생성
Superpowers 검증 중심 개발 절차를 담은 스킬 패키지 brainstorming, TDD, debugging, review를 강제하는 하네스 사례 요구사항 → 계획 → 테스트 → 구현 → 리뷰 흐름 해부
Subagents 별도 컨텍스트에서 일하는 전문 작업자 많은 파일 탐색, 리뷰, 보안 점검처럼 메인 컨텍스트를 오염시키는 일을 분리 리뷰어 에이전트가 변경점만 검토
Hooks 특정 이벤트마다 반드시 실행되는 자동화 지침이 아니라 강제 실행 레이어. lint, test, 위험 명령 차단에 적합 파일 수정 후 검사 메시지 출력 또는 테스트 실행

0:15-0:45

`CLAUDE.md` 설정 및 관리 꿀팁

“`CLAUDE.md`는 Claude가 매 세션 시작부터 보는 프로젝트 기억입니다. 그래서 여기에 너무 많은 것을 넣으면 똑똑해지는 게 아니라, 매 요청마다 불필요한 소음이 붙습니다. 항상 필요한 규칙만 넣고, 가끔 필요한 긴 문서는 Skill이나 별도 rules로 분리해야 합니다.” 
# CLAUDE.md 예시
## Project Rules
- Use pnpm, not npm.
- Before editing, inspect the target file.
- After code changes, run the narrowest relevant test.
- Do not commit unless the user explicitly asks.

## Verification
- TypeScript: pnpm typecheck
- Unit tests: pnpm test
- Lint: pnpm lint

## Architecture Notes
- API handlers live in src/server/routes.
- Shared UI components live in src/components.
- Do not call external APIs directly from UI components.
실습: 수강생 프로젝트 또는 샘플 프로젝트에 `CLAUDE.md`를 만들고, “항상 필요한 규칙”과 “가끔 필요한 참고자료”를 분리해 봅니다.

이미지 자료: Claude Code memory 문서의 메모리 위치 표를 캡처해 설명 자료로 사용. 출처: Claude Code memory docs.

Skills 구조: 필요할 때만 로드되는 작업 매뉴얼

“Skill은 `CLAUDE.md`를 대체하는 게 아닙니다. `CLAUDE.md`가 항상 켜져 있는 배경지식이라면, Skill은 특정 작업을 할 때 꺼내 쓰는 절차서입니다. 좋은 Skill의 핵심은 `description`입니다. Claude는 이 설명을 보고 언제 이 Skill을 써야 하는지 판단합니다.”

├── SKILL.md
├── examples/
│   └── good-triage.md
└── scripts/
    └── collect-context.sh
---
name: issue-triage
description: Analyze a bug report or feature request, identify affected files, risks, and verification steps. Use before implementing a small code change.
---

# Issue Triage

1. Restate the request in one sentence.
2. Find likely affected files.
3. Identify risks and edge cases.
4. Propose the smallest implementation plan.
5. List verification commands.
실습: `issue-triage` 스킬을 만들고, 모호한 사용자 요청을 넣어 분석 결과가 구조화되는지 확인합니다.

Superpowers 해부: 스킬인가, 하네스인가, 워크플로우인가

“Superpowers는 정확히 말하면 스킬 기반 하네스 엔지니어링 패키지입니다. 겉으로는 Skills 묶음이지만, 목적은 AI에게 개발자의 절차를 강제하는 것입니다. 요구사항 정리 없이 구현하지 말고, 실패 재현 없이 고치지 말고, 테스트 없이 완료라고 말하지 말라는 식의 작업 규율을 넣습니다.”
Superpowers 구성 강의 해부 포인트
brainstorming 구현 전 요구사항, 범위, 성공 기준을 정리하게 함
test-driven-development 실패하는 테스트 → 구현 → 리팩터링 순서를 강제
systematic-debugging 증상 추측이 아니라 재현, 원인 분석, 최소 수정으로 접근
requesting-code-review 완료 선언 전에 별도 검토 루프를 거치게 함
writing-skills 반복되는 절차를 재사용 가능한 Skill로 승격

이미지 자료: Superpowers 공식 플러그인 페이지의 소개 화면 또는 기능 목록을 캡처. 출처: Superpowers plugin page.

Subagents: 컨텍스트 격리와 역할 분리

“Subagent는 단순한 역할 프롬프트가 아닙니다. 별도 컨텍스트에서 읽고, 판단하고, 요약만 돌려주는 작업자입니다. 코드베이스 전체를 훑는 리뷰나 보안 점검을 메인 대화에서 직접 시키면 대화창이 금방 오염됩니다. 이때 Subagent를 씁니다.”

name: code-reviewer
description: Reviews local code changes for correctness, edge cases, and missing tests.
tools: Read, Grep, Glob, Bash
---

You are a pragmatic senior code reviewer.
Focus on bugs, regressions, missing tests, and risky assumptions.
Return findings first, ordered by severity, with file and line references.
데모: “Use a subagent to review this code for edge cases”처럼 명시적으로 호출하는 문장을 보여 줍니다.

2:35-3:10

Hooks: AI가 반드시 지나가는 자동화 검문소

“`CLAUDE.md`와 Skill은 모델이 읽는 지침입니다. 하지만 Hook은 이벤트가 발생했을 때 실제 명령을 실행합니다. ‘테스트를 돌려라’는 지침은 잊힐 수 있지만, PostToolUse hook은 파일 수정 뒤 항상 실행됩니다. 그래서 반드시 지켜야 하는 규칙은 프롬프트가 아니라 Hook으로 올려야 합니다.” 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'File changed. Run targeted verification.'"
          }
        ]
      }
    ]
  }
}
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "./scripts/block-dangerous-commands.sh"
          }
        ]
      }
    ]
  }
}
실습: 첫 단계는 `echo`로 시작합니다. 그 다음 프로젝트 언어에 맞춰 `pnpm lint`, `pytest`, `ruff check`, `go test ./...` 중 하나로 교체합니다.

이미지 자료: Hook vs Skill 비교 표를 슬라이드로 사용. 공식 문서의 “Hook은 lifecycle event, Skill은 Claude가 읽는 instruction” 비교가 근거입니다. 출처: Extend Claude Code, Hooks reference.

3:10-3:45

통합 실습: 작은 자동화 워크플로우 만들기

“이제 조각을 합칩니다. `CLAUDE.md`는 기본 규칙을 줍니다. Skill은 요청 분석 절차를 줍니다. Hook은 수정 뒤 검증을 강제합니다. Subagent는 리뷰를 분리합니다. 이 네 가지가 합쳐지면 에이전트가 자기 마음대로 코딩하는 것이 아니라, 우리가 만든 작업장 안에서 움직이게 됩니다.” 
목표 워크플로우

1. 사용자 요청 입력
2. /issue-triage 로 범위와 위험 분석
3. Claude가 최소 변경 구현
4. PostToolUse hook이 검사 명령 실행
5. code-reviewer subagent가 변경점 리뷰
6. Claude가 수정 사항과 검증 결과를 최종 보고
실습 과제: “버그 리포트 하나를 받아 원인 후보, 수정 계획, 검증 명령까지 자동으로 정리하는 workflow”를 만듭니다. 완성 기준은 파일 구조가 있고, 예시 요청을 넣었을 때 분석-구현-검증 보고 흐름이 나오는 것입니다.

정리와 Q&A

“오늘 만든 것은 거대한 자동화 시스템이 아닙니다. 하지만 구조는 실무와 같습니다. 반복되는 지시는 `CLAUDE.md`나 Skill로 올리고, 반드시 지켜야 하는 검증은 Hook으로 올리고, 복잡한 탐색과 리뷰는 Subagent로 분리합니다. 이게 AI 코딩 도구를 팀의 작업 방식에 맞추는 하네스 엔지니어링입니다.”

강의 품질 보완 포인트

초보자 보호 장치

  • 처음부터 테스트 실행 hook을 넣지 말고 `echo` hook으로 동작 원리를 확인합니다.
  • Hook은 실제 명령을 실행하므로 위험 명령 차단과 권한 개념을 먼저 설명합니다.
  • Skill은 길게 쓰기보다 `description`과 단계가 명확한지 검토하게 합니다.

중급자 확장 과제

  • `disable-model-invocation: true`로 수동 호출 전용 Skill을 만들어 컨텍스트 비용을 줄입니다.
  • 리뷰 전용 Subagent에 `tools`를 제한해서 읽기 중심 리뷰어로 만듭니다.
  • PostToolUse hook 결과를 Claude가 읽고 자동 수정하게 하는 루프를 구성합니다.
