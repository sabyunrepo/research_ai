00 · 타이틀

AI 에이전트 하네스 엔지니어링

프롬프트를 넘어 AI가 일하는 환경을 설계한다는 관점으로 시작합니다.

상세 발표 스크립트

오늘 강의의 제목은 AI 에이전트 하네스 엔지니어링입니다. 여기서 하네스라는 말은 AI를 묶어 둔다는 뜻이 아니라, AI가 안전하고 반복 가능한 방식으로 일하도록 주변 환경을 설계한다는 뜻입니다. 자동차 경주에서 운전자가 아무리 뛰어나도 안전벨트, 트랙, 신호, 정비 체계가 필요하듯이, AI 코딩 에이전트도 모델의 능력만으로는 충분하지 않습니다.

우리가 오늘 다룰 것은 프롬프트 하나를 잘 쓰는 요령에서 끝나지 않습니다. 프롬프트는 시작점입니다. 그 위에 프로젝트 기억을 담는 CLAUDE.md, 반복 절차를 담는 Skills, 개발 규율을 강제하는 Superpowers, 역할을 나누는 Subagents, 반드시 실행되는 Hooks, 외부 시스템을 연결하는 MCP, 그리고 완료를 증명하는 Evaluation을 쌓아 올릴 겁니다.

강의의 목표는 수강생이 끝나고 나서 “AI에게 이렇게 말하면 된다”가 아니라 “우리 팀의 AI 작업 환경은 이렇게 구성하면 된다”라고 말할 수 있게 만드는 것입니다. 그래서 마지막에는 작은 자동화 워크플로우를 하나 완성하는 방향으로 갑니다.

보강 포인트

프롬프트 강의가 아니라 작업 시스템 설계 강의라는 기대치를 초반에 확실히 잡습니다.

01 · 문제 정의

왜 하네스인가

AI는 똑똑하지만 일관되지 않으므로 주변 구조가 필요합니다.

상세 발표 스크립트

AI 코딩 도구를 써 보면 처음에는 굉장히 인상적입니다. 코드를 빠르게 읽고, 파일을 고치고, 테스트도 제안합니다. 그런데 실무에서 계속 써 보면 문제가 생깁니다. 어떤 날은 테스트를 잘 돌리는데, 어떤 날은 생략합니다. 어떤 요청에서는 범위를 잘 지키는데, 어떤 요청에서는 원하지 않은 리팩터링까지 섞습니다. 같은 팀 규칙을 매번 설명해야 하는 경우도 많습니다.

이 문제를 단순히 “프롬프트를 더 잘 써야 한다”로만 보면 한계가 있습니다. 프롬프트는 대화 한 번에는 효과가 있지만, 프로젝트 전체와 팀 전체의 반복적인 작업 습관을 보장하지는 못합니다. 그래서 우리는 지시를 여러 단계로 승격시켜야 합니다. 한 번만 필요한 말은 프롬프트에 두고, 반복되는 규칙은 CLAUDE.md에 두고, 반복되는 절차는 Skill로 만들고, 역할 분리가 필요하면 Subagent로 만들고, 반드시 실행되어야 하는 검증은 Hook으로 올립니다.

하네스 엔지니어링은 바로 이 승격 기준을 설계하는 일입니다. AI가 알아서 잘해 주기를 기대하는 것이 아니라, AI가 지나가는 길목마다 필요한 정보를 주고, 필요한 절차를 열고, 필요한 검증을 강제하는 구조를 만드는 것입니다.

보강 포인트

“프롬프트 → CLAUDE.md → Skill → Subagent → Hook” 승격 기준을 이른 시점에 반복합니다.

02 · 실패 패턴

실패는 절차 문제입니다

검증 누락, 컨텍스트 오염, 임의 구현을 먼저 보여 줍니다.

상세 발표 스크립트

AI가 실패하는 장면을 자세히 보면 꽤 반복적인 패턴이 있습니다. 첫 번째는 확인하지 않고 고치는 것입니다. 파일을 읽지 않았는데도 함수 이름을 추측해서 수정하거나, 기존 스타일을 보지 않고 새 패턴을 만들어 냅니다. 두 번째는 검증을 생략하는 것입니다. 테스트를 돌리지 않았는데 “수정했습니다”라고 말하거나, 테스트가 실패했는데 원인을 찾지 않고 skip이나 disable로 숨기려는 경우가 있습니다.

세 번째는 범위가 흐려지는 문제입니다. 사용자는 작은 버그 수정을 요청했는데, AI가 주변 구조를 크게 바꾸거나 관련 없는 리팩터링을 섞습니다. 네 번째는 컨텍스트 오염입니다. 대화가 길어지면서 앞의 실험, 중간 가정, 폐기된 요구사항이 섞이고, AI가 현재 목표와 다른 방향으로 움직입니다.

중요한 점은 이런 실패가 모델이 “나빠서”만 생기는 것이 아니라는 점입니다. 절차가 없으면 사람도 같은 실수를 합니다. 그래서 뒤에서 다룰 각 레이어는 이 실패 패턴에 대응합니다. 파일을 먼저 읽는 규칙은 CLAUDE.md에 넣고, 버그 분석 절차는 Skill로 만들고, 리뷰는 Subagent로 분리하고, 테스트 실행은 Hook으로 강제하고, 완료 기준은 Evaluation으로 확인합니다.

보강 포인트

실패 사례를 한두 개 실제 경험처럼 말하면 뒤의 기능들이 왜 필요한지 자연스럽게 연결됩니다.

03 · 전체 지도

워크플로우는 레이어로 봅니다

Prompt → Context → Skill → Agent → Hook → Tool → Evaluation → Loop.

상세 발표 스크립트

이제 오늘 강의 전체를 하나의 지도처럼 보겠습니다. 가장 안쪽에는 Prompt Layer가 있습니다. 여기에는 페르소나, few-shot 예시, 출력 형식, XML 태그 같은 즉석 지시가 들어갑니다. 그 다음은 Context Layer입니다. CLAUDE.md와 컨텍스트 엔지니어링이 여기에 속합니다. AI가 어떤 정보를 항상 기억하고, 어떤 정보는 필요할 때만 열어야 하는지 결정하는 레이어입니다.

그 다음은 Skill Layer입니다. 반복되는 업무 절차를 재사용 가능한 매뉴얼로 만드는 단계입니다. Superpowers는 이 Skill Layer를 활용해 개발 절차를 강제하는 좋은 사례입니다. 이후에는 Agent Layer가 나옵니다. Subagent와 Agent Teams를 통해 리뷰, 보안, 리서치 같은 역할을 분리합니다.

그 위에는 Automation Layer가 있습니다. Hooks가 대표적입니다. 지침이 아니라 실제 이벤트에 반응해 명령을 실행합니다. MCP는 Tool Layer로 볼 수 있습니다. GitHub, 브라우저, DB, Figma 같은 외부 시스템과 연결합니다. 마지막에는 Evaluation과 Loop가 있습니다. 결과를 검증하고, 필요하면 반복하거나 스케줄링합니다. 오늘 강의는 이 순서를 따라 하나씩 쌓아 올립니다.

04 · Prompt Layer

프롬프트도 하네스입니다

페르소나, 퓨샷, 출력 형식, XML 태그를 가장 작은 제어 장치로 설명합니다.

상세 발표 스크립트

프롬프트 엔지니어링은 하네스 엔지니어링과 경쟁하는 개념이 아닙니다. 오히려 하네스의 가장 작은 단위입니다. 한 번만 필요한 지시는 프롬프트에 둡니다. 예를 들어 “이번 답변은 표로 정리해줘”, “이 코드만 리뷰해줘”, “보안 관점만 봐줘” 같은 요청은 그 순간의 프롬프트로 충분합니다.

하지만 같은 말을 계속 반복하게 되면 문제가 달라집니다. 매번 “테스트 실패를 숨기지 마”, “파일을 읽고 수정해”, “리뷰에서는 칭찬보다 결함을 먼저 말해”라고 입력하고 있다면, 그건 프롬프트가 아니라 시스템 규칙이나 Skill로 승격해야 할 후보입니다. 반복되는 지시는 기억으로, 반복되는 절차는 Skill로, 반복되는 역할은 Subagent로, 반드시 실행되어야 하는 일은 Hook으로 올립니다.

프롬프트 레이어에서 중요한 것은 네 가지입니다. 첫째, 역할을 부여할 때는 이름보다 판단 기준을 줍니다. 둘째, few-shot은 지식 전달보다 출력 형식 고정에 씁니다. 셋째, XML 태그나 명확한 구분자로 instructions, context, examples, input을 나눕니다. 넷째, 최신 reasoning 모델에는 불필요한 “think step by step”보다 간결한 목표와 검증 기준이 더 잘 맞는 경우가 많습니다.

05 · Persona

페르소나는 판단 기준입니다

역할명보다 우선순위와 금지사항을 명확히 합니다.

상세 발표 스크립트

페르소나 부여는 여전히 유용합니다. 다만 “너는 시니어 개발자야”처럼 역할 이름만 주는 방식은 효과가 제한적입니다. 모델은 시니어 개발자라는 말에서 어느 정도 톤과 관점을 추론할 수 있지만, 우리가 원하는 판단 기준까지 항상 정확히 맞추지는 못합니다. 그래서 페르소나는 이름보다 우선순위가 중요합니다.

예를 들어 코드 리뷰를 시킬 때 “너는 시니어 코드 리뷰어다”라고만 말하면, 모델은 코드 스타일, 가독성, 칭찬, 개선 제안을 섞어서 줄 수 있습니다. 하지만 우리가 원하는 것이 결함 탐지라면 이렇게 말해야 합니다. “칭찬보다 결함을 먼저 말한다. 우선순위는 correctness bug, regression, missing test, risky assumption 순서다. 파일과 라인을 포함한다. 요약은 findings 뒤에 둔다.” 이렇게 하면 역할이 아니라 판단 기준이 생깁니다.

반복되는 페르소나는 더 이상 프롬프트로 둘 필요가 없습니다. 매번 코드 리뷰어 역할을 부여한다면 code-reviewer Subagent로 만들 수 있습니다. 매번 같은 리뷰 순서를 요구한다면 code-review Skill로 만들 수 있습니다. 이 컷의 핵심은 페르소나는 시작점이고, 반복되면 하네스의 상위 레이어로 승격해야 한다는 것입니다.

06 · Few-shot

Few-shot은 출력 모양을 고정합니다

좋은 예시와 나쁜 예시를 짧게 보여 줍니다.

상세 발표 스크립트

Few-shot prompting은 모델에게 새로운 지식을 많이 가르치는 용도라기보다, 우리가 원하는 답변의 모양을 보여 주는 데 강합니다. 특히 리뷰, 요약, 분류, 보고서처럼 출력 형식이 중요한 작업에서 효과가 큽니다. 좋은 예시 하나만 있어도 모델은 제목 길이, 세부 설명의 깊이, 파일 경로를 넣는 방식, 우선순위 표기 방식을 따라 하려는 경향이 있습니다.

좋은 few-shot은 길지 않아야 합니다. 예를 들어 코드 리뷰에서는 좋은 리뷰 예시와 나쁜 리뷰 예시를 하나씩 보여 주면 충분할 수 있습니다. 좋은 예시는 “[P1] 결제 실패 시 주문 상태가 pending으로 남습니다. 파일: src/payment.ts. 이유: catch 블록에서 rollback이 없습니다.”처럼 구체적입니다. 나쁜 예시는 “코드가 좀 복잡합니다”처럼 판단 기준도, 위치도, 영향도 없는 문장입니다.

주의할 점도 있습니다. 모든 요청에 few-shot을 붙이면 컨텍스트가 무거워집니다. 예시가 자주 반복된다면 프롬프트 안에 계속 붙이지 말고 Skill의 references로 옮기는 것이 좋습니다. 그러면 필요할 때만 예시가 열리고, 평소에는 Skill description만 가볍게 유지됩니다.

few-shot은 “많이”보다 “짧고 대표적인 예시”가 중요하다고 강조합니다.

07 · Reasoning Prompting

생각하라고 하기보다 조건을 선명하게

최신 reasoning 모델에는 간결한 목표와 검증 기준이 더 잘 맞습니다.

상세 발표 스크립트

예전에는 복잡한 문제를 시킬 때 “step by step으로 생각해” 같은 문구를 많이 붙였습니다. 지금도 어떤 상황에서는 도움이 될 수 있지만, 최신 reasoning 모델을 쓸 때는 습관처럼 붙이는 것이 항상 좋은 전략은 아닙니다. 모델이 이미 내부적으로 추론 능력을 갖고 있기 때문에, 우리가 더 잘해야 하는 일은 생각을 길게 하라고 말하는 것이 아니라 문제의 경계와 성공 기준을 선명하게 주는 것입니다.

예를 들어 “이 버그를 고쳐줘. 천천히 생각해”보다 “증상을 재현하고, 원인 후보를 세우고, 가장 작은 수정만 적용하고, 관련 테스트를 실행한 뒤, 변경 파일과 검증 결과를 보고해”가 더 실무적입니다. 이 문장은 모델의 생각 과정을 요구하지 않고, 작업 절차와 산출물을 지정합니다. 결과적으로 답변이 더 검증 가능해집니다.

이 컷에서 수강생에게 전달할 기준은 간단합니다. 먼저 zero-shot으로 명확한 목표와 입력 구분, 출력 형식을 줍니다. 부족하면 few-shot을 추가합니다. 모델의 내부 사고를 길게 보여 달라고 하기보다, 반례, 검증 명령, 남은 위험, 판단 기준을 요구합니다. 이것이 하네스 관점의 reasoning prompting입니다.

08 · CLAUDE.md

상세 발표 스크립트

이제 Prompt Layer에서 Memory Layer로 올라갑니다. CLAUDE.md는 Claude가 프로젝트에서 항상 참고하는 기억입니다. 팀이 매번 반복해서 설명하는 규칙을 여기에 넣습니다. 예를 들어 “파일을 수정하기 전에 기존 파일을 읽어라”, “테스트 실패를 skip하지 마라”, “커밋은 요청받을 때만 해라”, “이 프로젝트는 pnpm을 쓴다” 같은 규칙입니다.

하지만 CLAUDE.md는 길게 쓸수록 좋아지는 파일이 아닙니다. 항상 로드되는 기억이기 때문에, 여기에 너무 많은 내용을 넣으면 모든 요청에 불필요한 정보가 따라다닙니다. 그러면 모델이 중요한 규칙을 놓치거나, 컨텍스트 비용이 커지거나, 서로 충돌하는 지시가 생길 수 있습니다. 그래서 CLAUDE.md에는 항상 필요한 규칙만 넣어야 합니다.

좋은 기준은 이것입니다. “이 내용을 다음 20번의 요청에서도 거의 항상 알아야 하는가?” 그렇다면 CLAUDE.md 후보입니다. “가끔 특정 작업에서만 필요한가?” 그렇다면 Skill이나 reference 후보입니다. “실제로 반드시 실행돼야 하는가?” 그렇다면 Hook 후보입니다. 이렇게 나누면 CLAUDE.md가 프로젝트 헌법처럼 작동합니다.

요즘 AI 작업에서 자주 나오는 말이 컨텍스트 엔지니어링입니다. 프롬프트 엔지니어링이 이번 답변을 잘하게 만드는 기술이라면, 컨텍스트 엔지니어링은 여러 턴, 여러 파일, 여러 도구를 거치는 동안 AI가 필요한 정보를 잃지 않고 올바르게 일하게 만드는 설계입니다.

상세 발표 스크립트

Skill은 반복되는 작업 절차를 파일로 만든 것입니다. 예를 들어 코드 리뷰를 할 때마다 “버그를 먼저 보고, 파일과 라인을 적고, 테스트 누락을 확인해”라고 말하고 있다면, 그건 Skill이 될 수 있습니다. 버그 분석, 문서 작성, PR 요약, 스프레드시트 정리, 이미지 생성 같은 반복 작업도 모두 Skill 후보입니다.

Skill에서 가장 중요한 부분은 description입니다. 모델은 Skill 본문을 항상 읽지 않습니다. 먼저 이름과 description을 보고, 이 작업에 해당 Skill이 필요한지 판단합니다. 그래서 description은 짧지만 구체적이어야 합니다. “코드 리뷰를 한다”보다 “로컬 코드 변경에서 버그, 회귀, 테스트 누락을 우선순위로 리뷰할 때 사용한다”가 더 좋습니다.

Skill은 지식 창고가 아닙니다. 실행 매뉴얼입니다. 모델이 이미 아는 일반론을 길게 넣을 필요는 없습니다. 우리 조직, 우리 프로젝트, 이 작업에서만 중요한 절차를 넣어야 합니다. 그리고 긴 예시나 도메인 문서는 references로 빼고, 반복 실행이 필요한 로직은 scripts로 빼는 것이 좋습니다.

11 · Skill 구조

좋은 Skill은 작고 선명합니다

SKILL.md, references, scripts, assets의 역할을 나눕니다.

상세 발표 스크립트

Skill의 기본 구조는 단순합니다. 가장 중요한 파일은 SKILL.md입니다. 여기에는 name과 description이 들어가고, 본문에는 작업 절차가 들어갑니다. 하지만 모든 것을 SKILL.md에 넣으면 안 됩니다. SKILL.md가 너무 길어지면 Skill이 열릴 때마다 많은 컨텍스트가 들어오고, 모델이 핵심 절차를 놓칠 수 있습니다.

그래서 supporting files를 나눕니다. references는 필요할 때 읽을 문서입니다. 예를 들어 회사 API 규칙, 도메인 용어, 예시 출력 형식이 들어갈 수 있습니다. scripts는 반복해서 정확히 실행해야 하는 코드입니다. 예를 들어 파일 변환, 데이터 검증, 템플릿 생성처럼 모델이 매번 코드를 새로 짜면 위험한 작업에 좋습니다. assets는 산출물에 사용할 템플릿, 이미지, 예제 파일입니다.

좋은 Skill은 모델에게 모든 것을 설명하려고 하지 않습니다. 모델은 이미 똑똑하다는 전제를 둡니다. 그래서 “이 작업에서는 어떤 순서로 움직여야 하는가”, “무엇을 확인해야 하는가”, “어떤 파일을 필요할 때 열어야 하는가”만 선명하게 적습니다. 이 구조를 이해하면 수강생이 자기 팀의 반복 업무를 Skill로 바꾸기 쉬워집니다.

15 · Agent Teams

병렬 에이전트는 관점을 분리합니다

security, performance, test 관점을 동시에 돌릴 수 있습니다.

상세 발표 스크립트

Subagent가 하나의 역할 분리라면, Agent Teams는 여러 관점을 병렬로 돌리는 방식입니다. 모든 작업에 필요한 것은 아니지만, 복잡한 변경에서는 매우 유용합니다. 한 에이전트는 구현을 하고, 다른 에이전트는 보안 관점에서 위험을 보고, 또 다른 에이전트는 테스트 누락을 보고, 또 다른 에이전트는 성능 영향을 볼 수 있습니다.

다만 병렬 에이전트는 아무 때나 쓰면 오히려 혼란이 생깁니다. 같은 파일을 여러 에이전트가 동시에 고치면 충돌이 생깁니다. 그래서 범위를 명확히 나눠야 합니다. 예를 들어 한 에이전트는 읽기 전용 리뷰만 하고, 다른 에이전트는 문서만 수정하고, 또 다른 에이전트는 테스트만 추가하는 식입니다. 더 큰 프로젝트에서는 git worktree를 써서 작업 공간 자체를 분리할 수도 있습니다.

강의에서는 이 부분을 고급 확장으로 설명하면 좋습니다. 처음에는 단일 에이전트와 리뷰 Subagent만으로도 충분합니다. 하지만 팀 규모가 커지고, 검증 관점이 다양해지면 Agent Teams가 실전적인 패턴이 됩니다. 핵심은 병렬화가 아니라 책임 분리입니다.

16 · Hooks

Hook은 지시가 아니라 실행입니다

파일 수정, 권한 요청, 세션 시작, 종료 이벤트에 명령을 연결합니다.

상세 발표 스크립트

이제 지침에서 실행으로 넘어갑니다. CLAUDE.md와 Skill은 모델이 읽는 지침입니다. 모델이 잘 따르면 좋지만, 결국 지침입니다. 반면 Hook은 특정 이벤트가 발생했을 때 실제 명령을 실행하는 장치입니다. 예를 들어 파일을 수정한 직후 formatter나 test를 실행하게 만들 수 있습니다.

이 차이는 실무에서 큽니다. “수정 후 테스트를 돌려라”라고 CLAUDE.md에 적어 두면 모델이 따를 가능성이 높아집니다. 하지만 상황에 따라 놓칠 수도 있습니다. PostToolUse Hook에 테스트 명령을 연결하면 파일 수정 뒤 실제로 명령이 실행됩니다. 지침이 아니라 검문소가 되는 것입니다.

처음부터 무거운 Hook을 만들 필요는 없습니다. 강의 실습에서는 echo로 시작하는 것이 좋습니다. 파일 수정 후 “검증을 실행하세요”라는 메시지를 출력하는 Hook을 먼저 만들고, 동작 원리를 이해한 다음 lint나 test로 바꿉니다. Hook은 강력한 만큼 위험 명령, 긴 실행 시간, 권한 범위를 함께 고려해야 합니다.

17 · Hook 심화

Hook도 단계가 있습니다

Stop Hook, Prompt Hook, Agent Hook으로 검증을 강화합니다.

상세 발표 스크립트

Hook을 단순히 “파일 수정 후 테스트 실행” 정도로만 이해하면 절반만 본 것입니다. Hook은 여러 단계로 확장할 수 있습니다. SessionStart Hook은 세션이 시작될 때 동적 컨텍스트를 불러올 수 있습니다. PostToolUse Hook은 파일 수정 뒤 formatter, lint, test를 실행할 수 있습니다. PermissionRequest Hook은 위험한 권한 요청을 외부 승인 흐름으로 보낼 수 있습니다.

특히 Stop Hook은 강의에 넣을 가치가 큽니다. AI가 “작업 끝”이라고 말하기 전에 검증이 되었는지 확인하고, 조건이 맞지 않으면 종료를 막을 수 있습니다. Prompt Hook은 간단한 판단을 프롬프트로 처리하는 방식이고, Agent Hook은 코드베이스를 읽어야 하는 복잡한 검증을 별도 에이전트에게 맡기는 방식입니다.

이 부분의 핵심은 Hook도 성숙도가 있다는 것입니다. 첫 단계는 command hook입니다. 정해진 명령을 실행합니다. 다음은 prompt hook입니다. 완료 조건을 판단합니다. 그 다음은 agent hook입니다. 별도 컨텍스트에서 검토합니다. 이렇게 가면 하네스가 단순 자동 실행에서 판단 자동화로 발전합니다.

보강 포인트

18 · MCP / Tool Layer

MCP는 실제 세계와 연결합니다

GitHub, DB, Slack, Browser, Figma 같은 도구 연결 레이어입니다.

상세 발표 스크립트

지금까지는 AI가 어떻게 생각하고, 기억하고, 절차를 따르고, 검증을 통과하게 할지 이야기했습니다. 이제 외부 세계와 연결하는 레이어가 필요합니다. MCP는 AI 에이전트가 GitHub, 데이터베이스, Slack, 브라우저, Figma 같은 도구와 연결되는 방식입니다. Skill이 도구 사용법을 가르친다면, MCP는 실제 도구를 사용할 수 있게 해 줍니다.

예를 들어 GitHub MCP가 있으면 이슈와 PR을 읽을 수 있습니다. Browser MCP가 있으면 로컬 웹 UI를 실제로 열어 확인할 수 있습니다. DB MCP가 있으면 운영 데이터를 조회할 수 있습니다. Figma MCP가 있으면 디자인 문맥을 가져올 수 있습니다. 이렇게 되면 AI는 단순 코드 생성기가 아니라 작업 환경 안의 실행자가 됩니다.

하지만 MCP를 많이 붙이는 것이 항상 좋은 것은 아닙니다. 도구 설명도 컨텍스트를 차지하고, 쓰기 권한이 있는 도구는 위험을 만듭니다. 그래서 필요한 도구만 연결하고, 읽기와 쓰기 권한을 분리하고, 위험한 작업은 Hook이나 사람 승인으로 감싸야 합니다. Tool Layer 역시 하네스 설계의 일부입니다.

보강 포인트

“Skill은 사용법, MCP는 실제 연결”이라는 문장으로 구분합니다.

19 · Evaluation

완료의 기준은 검증입니다

test, lint, judge, human review를 함께 둡니다.

상세 발표 스크립트

AI가 “완료했습니다”라고 말하는 것은 완료의 증거가 아닙니다. 사람 개발자도 마찬가지입니다. 완료의 기준은 검증입니다. 기계가 확인할 수 있는 것은 기계로 확인해야 합니다. 테스트, 린트, 타입체크, 빌드가 여기에 해당합니다. 이것들은 Hook으로 자동 실행하거나, 최종 보고 전에 반드시 실행하게 만들 수 있습니다.

하지만 모든 품질이 테스트로만 확인되지는 않습니다. 요구사항을 제대로 반영했는지, 사용자 경험이 자연스러운지, 리뷰 관점에서 위험한 가정이 있는지는 판단이 필요합니다. 이때 Subagent 리뷰어, LLM-as-judge, rubric 기반 평가, 사람 리뷰를 조합할 수 있습니다. 중요한 것은 평가 기준을 사전에 정하는 것입니다.

강의 실습의 최종 보고 형식도 이 원칙을 따라야 합니다. “무엇을 바꿨는가”, “어떤 파일을 수정했는가”, “어떤 검증 명령을 실행했는가”, “검증 결과는 무엇인가”, “남은 위험은 무엇인가”를 말하게 합니다. 이렇게 하면 AI의 완료 선언을 믿는 것이 아니라, 완료를 증명하는 구조가 생깁니다.

20 · Loop / Schedule

자동화는 한 번으로 끝나지 않습니다

반복 감시, 주기 점검, 문서 업데이트로 확장합니다.

상세 발표 스크립트

지금까지의 흐름은 주로 한 번의 작업 요청을 기준으로 설명했습니다. 하지만 실무 자동화는 한 번 실행하고 끝나는 경우보다 계속 확인하고 반복해야 하는 경우가 많습니다. 예를 들어 PR 상태를 주기적으로 확인하거나, 실패한 CI 로그가 올라오면 다시 분석하거나, 매일 변경된 문서를 업데이트하거나, 특정 지표가 바뀌면 보고서를 만드는 작업입니다.

여기서 Hook, Loop, Schedule을 구분하면 좋습니다. Hook은 이벤트 기반입니다. 파일이 수정되거나 권한 요청이 들어오거나 세션이 끝나려 할 때 실행됩니다. Loop는 조건이 만족될 때까지 반복하는 구조입니다. Schedule은 시간 기반입니다. 매일 아침, 매주 월요일, 30분마다 같은 규칙으로 작업을 실행합니다.

이 부분은 고급 확장입니다. 처음부터 모든 것을 자동화하려고 하면 위험합니다. 먼저 작은 워크플로우를 만들고, 검증 기준을 만들고, 그 다음 반복 가능한 작업을 Schedule로 올립니다. 이렇게 하면 AI 에이전트는 단순 코딩 도구에서 운영 보조자, 품질 감시자, 문서 유지 관리자로 확장됩니다.

보강 포인트

21 · 통합 실습

최종 산출물은 작은 작업장입니다

CLAUDE.md, Skill, Subagent, Hook, Evaluation을 하나로 연결합니다.

상세 발표 스크립트

마지막으로 오늘 배운 내용을 하나의 작은 작업장으로 묶습니다. 거대한 플랫폼을 만들 필요는 없습니다. 수강생이 가져갈 수 있는 최소 산출물은 명확합니다. 첫째, 프로젝트 규칙을 담은 CLAUDE.md입니다. 둘째, 요청을 분석하는 Skill입니다. 셋째, 코드 리뷰를 분리하는 Subagent입니다. 넷째, 수정 후 검증을 강제하는 Hook입니다. 다섯째, 최종 보고에 들어갈 Evaluation 기준입니다.

실습 흐름은 이렇게 잡으면 됩니다. 사용자가 버그 수정 요청을 넣습니다. 먼저 Prompt Layer에서 요청을 한 문장으로 정리합니다. CLAUDE.md의 규칙에 따라 파일을 먼저 읽습니다. issue-triage Skill로 영향 범위와 위험을 정리합니다. 구현은 최소 변경으로 합니다. PostToolUse Hook이 검증을 실행하거나 검증 알림을 띄웁니다. reviewer Subagent가 변경점을 검토합니다. 마지막 보고에는 변경 파일, 검증 명령, 검증 결과, 남은 위험이 들어갑니다.

마무리 질문은 이것입니다. “우리 팀에서 AI가 두 번 이상 틀린 규칙은 무엇인가?” 그 답이 다음 CLAUDE.md 항목, 다음 Skill, 다음 Subagent, 다음 Hook 후보입니다. 이 질문을 팀 회고에 넣으면 AI 사용 경험이 단순한 개인 노하우가 아니라 팀의 작업 시스템으로 축적됩니다.
